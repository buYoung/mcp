## The mechanism: `dictGetOrDefault` with a tuple of attributes

The "enrichment from a reference source" is `dictGetOrDefault('dict', ('a', 'b', 'c'), key, default_tuple)`. The consumer wraps it in `tupleElement(..., 1)` to read only attribute `a`. Engineers then place garbage values in default slots 2 and 3, assuming those slots are never evaluated.

### Why unused fallback slots can still cause a throw

In `FunctionsExternalDictionaries.h:483`, `executeImpl` casts **the entire default tuple** before splitting it into per-attribute columns:

```cpp
// line 481-499
ColumnWithTypeAndName column_to_cast = {column_before_cast.column->convertToFullColumnIfConst(), ...};
auto result = castColumnAccurate(column_to_cast, result_type);   // ← whole tuple cast here

if (attribute_names.size() > 1)
{
    const auto * tuple_column = checkAndGetColumn<ColumnTuple>(result.get());
    ...
    default_cols = tuple_column->getColumnsCopy();   // ← split AFTER cast
}
```

`castColumnAccurate` is called on the **full** default tuple, casting every element to its corresponding dictionary attribute type. If slot 2 or 3 holds a `String` where a `UInt64` attribute is expected, that cast throws — regardless of the fact that `tupleElement` will only expose slot 1 to the caller. The exception fires before the tuple is ever split.

This is explicitly acknowledged in `DictGetTupleElementPass.cpp:166–170` (the optimizer's own safety comment):

> `dictGetOrDefault` casts the entire default tuple to the dict's attribute tuple type at execution. If the element types of the default tuple don't match the dict's attribute types, that cast can throw on an un-selected element.

### Why a one-field form surfaces a different type

The optimizer (`DictGetTupleElementPass`) tries to rewrite  
`tupleElement(dictGet('dict', ('a','b','c'), key), 1)` → `dictGet('dict', 'a', key)`

but the single-attribute form behaves differently with `LowCardinality`. From `DictGetTupleElementPass.cpp:247–251`:

> The single-attribute form of `dictGet` **preserves a `LowCardinality` wrapper** from the key argument, while the tuple-attribute form **drops it** (a tuple cannot be `LowCardinality`). If the result types differ, the rewrite would change the type observed by the parent node and break the surrounding query tree.

So `dictGet('dict', 'a', key)` may return `LowCardinality(String)` while the equivalent tuple subscript returns `String`. The optimizer bails out precisely because of this (line 250: `if (!new_dict_get_function.getResultType()->equals(*tuple_element_function->getResultType())) return;`), but without that guard the rewritten plan would expose a different type to every node above it in the query tree.

### Summary

Discarded fallback slots remain observable for two distinct reasons:

1. **Throw path**: `castColumnAccurate` is applied to the *whole* default tuple atomically at runtime, before any element is extracted. A type-incompatible "junk" value in an unread slot causes an exception even though nothing downstream ever reads that slot.

2. **Type-change path**: Converting the multi-attribute tuple form to the single-attribute form changes the result type when `LowCardinality` is involved — the tuple form always strips the wrapper, the scalar form preserves it — so the rest of the query plan sees a different type than the tuple form would have produced.
