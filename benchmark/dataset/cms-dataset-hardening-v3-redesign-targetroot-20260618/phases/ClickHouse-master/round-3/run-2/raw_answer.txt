The answer is entirely within `FunctionsExternalDictionaries.h`. There are two distinct mechanisms, both rooted in `FunctionDictGetNoType` (`dictGet` / `dictGetOrDefault`).

---

## Mechanism 1 — The fallback column is cast to the *full* result type before any field is extracted

`dictGetOrDefault('dict', ('attr1','attr2'), key, fallback_tuple)` — the multi-attribute, tuple-returning variant.

In `executeImpl` (line ~476–504), when the default argument is **not** a lazy/short-circuit column, this happens:

```cpp
// FunctionsExternalDictionaries.h ~482-499
ColumnWithTypeAndName column_to_cast = {
    column_before_cast.column->convertToFullColumnIfConst(),
    column_before_cast.type,
    column_before_cast.name};

auto result = castColumnAccurate(column_to_cast, result_type);   // ← entire tuple cast

if (attribute_names.size() > 1)
{
    const auto * tuple_column = checkAndGetColumn<ColumnTuple>(result.get());
    // ...
    default_cols = tuple_column->getColumnsCopy();    // ← then unpacked per-attribute
}
```

`result_type` is `DataTypeTuple(T1, T2, …)` — the type of **all** requested attributes. `castColumnAccurate` calls `CAST(fallback AS 'Tuple(T1,T2,...)')` under `CastType::accurate`, which validates every element individually. If any "sentinel junk" element in the fallback tuple is type-incompatible with its slot (e.g., a string `'xxx'` for a `UInt64` attribute), the cast throws — **regardless of which tuple element the caller ever reads downstream**. The fallback is validated as a unit before it is split.

---

## Mechanism 2 — Single-attribute vs. multi-attribute form surfaces a structurally different type to the plan

In `getReturnTypeImpl` (lines ~404–414):

```cpp
if (attribute_types.size() > 1)
{
    // multi-attribute path
    return std::make_shared<DataTypeTuple>(attribute_types, attribute_names);
}
// single-attribute path
if (key_is_nullable)
    return makeNullable(attribute_types.front());
return attribute_types.front();   // bare T, not Tuple(T)
```

- `dictGet('d', 'col1', k)` → returns bare **`T1`**  
- `dictGet('d', ('col1', 'col2'), k)` → returns **`Tuple(T1, T2)`**

These are different `DataTypePtr` objects. Everything built on top of that call node — `ORDER BY`, `GROUP BY`, type-checking in joins, column binding in projections — sees the declared return type during the planning phase. Switching from the one-string form to the tuple-of-strings form changes the advertised type even if the caller only ever accesses `.col1`, because type inference runs once at planning time on the whole expression, not field-by-field.

---

## Summary

The discarded fallback slots remain observable for two reasons, both structural:

1. **`dictGetOrDefault` with multiple attributes casts the entire fallback tuple to the full `DataTypeTuple` result type via `castColumnAccurate` at execution time**, before the tuple is split into per-attribute columns. A junk value in any slot triggers the accurate-cast type check and throws, independent of which slot the caller reads.

2. **`getReturnTypeImpl` returns `DataTypeTuple(T1,T2,…)` for the multi-attribute form and bare `T` for the single-attribute form**. That type is fixed at plan time and propagates to every node above the function. Moving junk into the second argument slot of the tuple name list changes the declared return type seen by the rest of the plan, even if no downstream code reads that field.
