**The rule**: Angular defers re-applying the model via `_queuedWrite` + `afterNextRender`, and filters child mutations with an `updateMode` gate.

**Why batching matters** (lines 153-170 of `select_control_value_accessor.ts`):

1. **Exponential `compareValue` calls**: Calling `writeValue` per individual option add/remove causes O(n²) equality checks as each operation re-evaluates all options.

2. **Safari/IE deselection bug**: Angular would set `select.value` *before* appending an option's text node, so the browser would ignore the model value and drift to the first option.

3. **Animation module removal delay**: When a selected option is removed, the browser would auto-select the first remaining option even though no model value matched—because the DOM update was still pending.

The `MutationObserver` in `observeSelectMutations()` (select.ts:29-43) coalesces child mutations, but the actual `writeValue` call is queued once via `afterNextRender` (select_control_value_accessor.ts:180-191). The `updateMode` flag in `control_native.ts:88` additionally prevents processing mutations before Angular's first update pass completes, since the initial render already sets `input.value` correctly.
