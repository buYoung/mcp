## 핵심: 모든 컬럼이 검증 대상입니다

### 1. 행 개수 검증 — `checkNumberOfRows()` (Block.cpp:420)

```cpp
void Block::checkNumberOfRows(bool allow_null_columns) const
{
    ssize_t rows = -1;
    for (const auto & elem : data)  // ← "used" 여부와 무관하게 ALL 컬럼 순회
    {
        // ...
        ssize_t size = elem.column->size();
        if (rows != size)
            throw Exception(ErrorCodes::SIZES_OF_COLUMNS_DOESNT_MATCH, ...);
    }
}
```

소비자가 **단 하나의 필드만 읽더라도**, Block은 **모든 컬럼**의 행 개수가 일치하는지 검증합니다. Sentinel junk를 채운 fallback 컬럼이 다른 행 개수를 가지면 `SIZES_OF_COLUMNS_DOESNT_MATCH` 예외가 발생합니다.

### 2. 구조 비교 — `checkBlockStructure()` (Block.cpp:133)

블록 비교 시 **위치 기반**으로 모든 컬럼을 비교합니다:

```cpp
for (size_t i = 0; i < columns; ++i)
{
    const auto & actual = lhs.getByPosition(i);   //_positions_로 접근
    const auto & expected = rhs.getByPosition(i);
    checkColumnStructure(actual, expected, ...);   // 이름+타입 검증
}
```

실제론 `checkColumnStructure` (Block.cpp:66:~76)에서 **타입 동등성**까지 검증하므로, 읽히지 않는 컬럼의 잘못된 타입이plan 전체에 영향을 줄 수 있습니다.

### 3. 타입 전파 — `getDataTypes()` (Block.cpp:775)

```cpp
DataTypes Block::getDataTypes() const
{
    for (const auto & elem : data)
        res.push_back(elem.type);  // ← 모든 컬럼의 타입 포함
    return res;
}
```

## 요약

| 검증 | 범위 | fallback 필드가 관찰되는 이유 |
|---|---|---|
| `checkNumberOfRows` | ALL 컬럼 | 읽히지 않는 컬럼도 행 수 검증 대상 |
| `checkBlockStructure` | ALL 컬럼 | 위치 기반 비교 — 사용 여부 무관 |
| `getDataTypes` | ALL 컬럼 | 쿼리 플랜의 타입 추론에 전체 구조 반영 |

**Discarded fallback fields remain observable**，是因为 Block 是一个**原子적 검증 단위** — 개별 컬럼이 아닌 블록 전체가 한번에 검증되고, 이 검증에는 이름·타입·행 개수가 포함됩니다. 사용자가 `SELECT a FROM enriched_table`로 `a`만 읽어도, `b`, `c` 같은 fallback 필드가 여전히 행 수 불일치나 타입 불일치 시 예외나 타입 전파에 관여합니다.
