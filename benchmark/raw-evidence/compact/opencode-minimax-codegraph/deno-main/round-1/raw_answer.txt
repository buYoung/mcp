**Resolution path that keeps packaging working:**

The `collect_bundle_required_packages` function in `cli/standalone/native_addons.rs:113` uses **longest-prefix matching** on `referenced_paths` to attribute file paths to owning npm packages. When a helper module (like `tests/specs/bundle/multiple_entries/src/bar/helper.ts`) pulls in files from neighboring source trees, the system walks all installed package folders and finds the deepest folder that is a prefix of each file path (`lines 160-184`). This ensures the correct owning package gets seeded for the BFS dependency closure that follows.

**Case where it can bind the wrong version:**

The `dedup_equivalent_peer_variants()` method in `libs/npm/resolution/snapshot.rs:196` merges packages with identical name+version and dependency signature. However, when a helper's bare specifier import (e.g., `import "foo"`) resolves at runtime to a package version that differs from what the **longest-prefix path attribution** would select, the embedded snapshot may contain one version while the actual node_modules directory available at runtime provides another.

This manifests when:
1. A helper module from a neighboring source tree imports a package via bare specifier
2. Multiple versions of that package exist in the workspace (e.g., hoisted `node_modules/pkg` vs nested `node_modules/org/pkg`)
3. Path-based attribution picks the owning package based on folder depth, not version correctness
4. The BFS closure then embeds a version that doesn't match what the import actually resolved to during graph building

The version that gets embedded is whoever "wins" in the snapshot's deduplication, but the runtime require/import might bind to a different folder's version if the workspace layout changed between snapshot creation and runtime.
