현재 이 흐름이 계속 동작하는 이유는 `deno compile --bundle`의 번들 해석기가 1차 해석 실패 뒤에 npm 스냅샷 폴백을 한 번 더 타기 때문입니다.

구체적으로는 [`cli/tools/bundle/mod.rs`](<REPO_ROOT>/.agents/benchmark-data/deno-main/cli/tools/bundle/mod.rs:1562)에서, 엔트리의 패키지 스코프 밖에서 끌려온 파일 때문에 bare import 해석이 실패하면 `resolve_bare_specifier_in_npm_snapshot(...)`를 호출합니다. 이 함수는 [`libs/resolver/graph.rs`](<REPO_ROOT>/.agents/benchmark-data/deno-main/libs/resolver/graph.rs:404)에서 bare specifier를 `npm:{name}`로 바꿔 사실상 `name@*` 요청으로 만든 뒤, [`resolve_managed_npm_req_ref`](<REPO_ROOT>/.agents/benchmark-data/deno-main/libs/resolver/graph.rs:371)를 통해 managed npm snapshot에서 패키지 폴더를 찾고, 거기서 다시 서브패스를 해석합니다. 실제 폴더 선택은 [`libs/resolver/npm/managed/mod.rs`](<REPO_ROOT>/.agents/benchmark-data/deno-main/libs/resolver/npm/managed/mod.rs:216)와 [`libs/resolver/npm/managed/resolution.rs`](<REPO_ROOT>/.agents/benchmark-data/deno-main/libs/resolver/npm/managed/resolution.rs:91)에서 snapshot의 `resolve_pkg_from_pkg_req(req)`로 끝납니다. 질문의 시나리오는 아예 테스트로 고정돼 있고, 설명도 [`tests/specs/compile/bundle/worker_npm_outside_scope/entry/main.ts`](<REPO_ROOT>/.agents/benchmark-data/deno-main/tests/specs/compile/bundle/worker_npm_outside_scope/entry/main.ts:1)에 적혀 있습니다.

조용히 잘못된 버전에 바인딩될 수 있는 경우도 현재 코드에 주석으로 명시돼 있습니다. [`libs/resolver/graph.rs`](<REPO_ROOT>/.agents/benchmark-data/deno-main/libs/resolver/graph.rs:417) 기준으로, 이 폴백은 버전 제약 없이 `name@*`로만 해석하므로 빌드의 npm snapshot 안에 같은 패키지의 여러 버전이 있으면 referrer 주변 `node_modules` 트리가 골랐어야 할 “가장 가까운 버전”이 아니라 snapshot이 이름만 보고 고른 다른 버전에 묶일 수 있습니다. 즉, 단일 버전일 때는 안전하지만, 다중 버전 공존 시에는 Node/Bun의 근접 해석과 달라질 수 있습니다.
