이제 핵심 파일들을 직접 확인해서 두 가지 관찰 가능성을 정확히 파악하겠습니다.
이제 `candidate_from_doc` 함수와 쿼리 파싱의 오류 전파 경로를 확인하겠습니다.
이제 두 가지 관찰 가능성을 정확히 짚겠습니다.

---

## 폴백 슬롯이 쿼리 에러를 관찰 가능하게 만드는 두 메커니즘

### 1. "미사용" 폴백 슬롯이 쿼리 throw 여부를 바꾸는 이유

`candidate_from_doc`(`ranking.rs:299-337`)은 BM25 점수만으로도 충분한 소비자에게 `extracted_json` 스토어 필드를 **무조건 읽고 역직렬화**합니다:

```rust
let extracted_json = doc
    .get_first(handle.extracted_json_field)   // ← 스토어 필드 읽기
    .and_then(|v| v.as_str())
    .unwrap_or("{}");

let extracted_file: ExtractedFile =
    serde_json::from_str(extracted_json)       ← 역직렬화 시도
        .unwrap_or_else(|_| ExtractedFile { … });  // ← 실패 시 빈 심볼/리터럴
```

소비자가 `score`나 `file_path`만 사용해도, `searcher.doc()`(`ranking.rs:306-308`)의 실패는 `?` 연산자로 **전체 검색을 에러로 전파**합니다:

```rust
let doc = searcher
    .doc::<TantivyDocument>(doc_address)
    .map_err(|e| e.to_string())?;   // ← 여기서 throw → 검색 전체 실패
```

즉 `extracted_json`이라는 "보강용 슬롯"이 깨지거나 문서가 손상되면, 점수에만 관심 있는 소비자까지 에러를 받게 됩니다. 이 슬롯을 읽지 않으면(=보강 없이) 점수와 경로만 안전하게 반환할 수 있습니다.

### 2. 단일 부울 필드가 하위 계획에 다른 "타입"을 노출하는 이유

`symbol_fallback`(`engine.rs:32`)은 단순 부울이지만, `matched_symbols`의 **의미적 타입**을 완전히 바꿉니다:

```rust
let symbol_fallback = matched_symbols.is_empty();
if symbol_fallback {
    matched_symbols = all_symbols;  // ← 매칭된 심볼 → 파일 전체 심볼
}
```

- `symbol_fallback = false`: `matched_symbols` = 쿼리와 **실제 매칭된** 심볼만
- `symbol_fallback = false`: `matched_symbols` = 파일의 **모든** 심볼

Rust 타입은 둘 다 `Vec<ExtractedSymbol>`이지만, **의미적 타입**이 다르고 렌더러/어노테이션 시스템이 이를 기준으로 분기합니다(`search/mod.rs:505-531`):

```rust
if res.symbol_fallback {
    // 이름+범위만 렌더링 (스니펫 없음), 바이트 예산 소비
    const FALLBACK_SNIPPET_CAP: usize = 5;
    …
}
```

소비자가 `symbol_fallback`을 읽지 않고 `matched_symbols`의 길이나 내용만으로 판단하면, 전체 심볼 목록을 "매칭된 심볼"로 착각하고 **스니펫 생성·바이트 예산을 과도하게 소비**합니다. 단일 부울이 렌더링 파이프라인의 분기 조건이 되어, 동일한 `Vec<ExtractedSymbol>` 타입이 서로 다른 실행 경로를 타게 만듭니다.

---

**요약**: 이 코드베이스에서 폴백 슬롯은 "읽지 않으면 무해"가 아니라 **에러 전파 경로에 연결**되어 있고, `symbol_fallback` 같은 단일 부울은 **의미적 타입을 전환**하는 스위치로 작동합니다. 그래서 엔지니어들이 "아무도 안 읽는 슬롯에 sentinel을 넣어도 된다"는 가정이 틀립니다.
