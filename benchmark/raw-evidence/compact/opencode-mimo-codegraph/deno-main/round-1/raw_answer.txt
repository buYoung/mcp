## 해석 경로 (Resolution Path)

이 코드베이스에서 해당 패키징 흐름이 작동하는 핵심 메커니즘은 **Node.js 모듈 해석 알고리즘이 디렉토리 트리를 위로 순회하는 것**입니다.

TypeScript 설정(`packages/typescript-config/base.json:10-12`)에서 `moduleResolution: "NodeNext"`이 선언되어 있으므로, 런타임에 Node.js의 네이티브 ESM 해석 규칙이 적용됩니다. `tsc`가 `src/index.ts`를 `dist/index.js`로 컴파일한 뒤, 시작 파일이 `dist/`에 존재하면 Node.js는 bare import(예: `import ... from "@modelcontextprotocol/sdk/server/mcp.js"`)를 다음과 같이 해석합니다:

```
dist/index.js
  → dist/node_modules/@modelcontextprotocol/sdk  ← 없음
  → apps/scout/node_modules/@modelcontextprotocol/sdk  ← 심볼릭 링크로 존재 ✓
```

pnpm의 `node_modules` 구조를 보면 각 앱은 자신의 선언된 의존성을 자기 `node_modules/`에 심볼릭 링크로 가지고 있습니다:

- `apps/scout/node_modules/@modelcontextprotocol/sdk/` → `.pnpm/` 스토어의 특정 변형으로 연결
- `apps/scout/node_modules/zod/` → `zod@4.4.3`으로 연결
- `apps/scout/node_modules/smol-toml/` → `smol-toml@1.6.1`로 연결

따라서 "인접 소스 트리에서 가져온 헬퍼 모듈"이 `dist/`에 컴파일되어 배포본에 포함되면, 해당 헬퍼의 bare import는 **배포 대상 앱의 `node_modules/`**를 통해 해석됩니다. 자기 `package.json`에 해당 의존성이 명시되지 않았더라도, Node의 상위 디렉토리 순회가 앱 레벨의 `node_modules/`를 찾아내기 때문에 빌드가 성공합니다.

## 조용히 잘못된 버전을 바인딩하는 경우

pnpm의 `.pnpm/` 스토어 디렉토리를 보면 `@modelcontextprotocol/sdk@1.29.0`이 **동시에 두 개의 변형**으로 존재합니다(`node_modules/.pnpm/`):

| 변형 | peer dependency | 사용자 |
|------|----------------|--------|
| `@modelcontextprotocol+sdk@1.29.0_zod@3.25.76/` | `zod@3.25.76` | `@modelcontextprotocol/inspector` (devDependency) |
| `@modelcontextprotocol+sdk@1.29.0_zod@4.4.3/` | `zod@4.4.3` | `apps/mcp-server`, `apps/scout` |

`@modelcontextprotocol/sdk`의 `package.json`(`pnpm-lock.yaml:596-604`)에서 peer dependency가 `zod: ^3.25 || ^4.0`으로 선언되어 있으므로, pnpm은 의존성 그래프 전체에서 각 소비자마다 올바른 peer 조합을 매핑합니다.

**잘못된 버전 바인딩이 발생하는 조건:**

배포본을 만들 때 pnpm의 심볼릭 링크 구조를 평탄화(flat `node_modules/`)하거나, 의존성 트리를 재구성하면 문제는 빚어집니다. `@modelcontextprotocol/sdk`의 두 변형은 동일한 `version: "1.29.0"`을 가지므로, 심볼릭 링크가 제거된 평탄화된 `node_modules/`에서는 **어느 변형이 노출될지 불확실**해집니다.

특히 `apps/scout/dist/`에 포함된 헬퍼가 `@modelcontextprotocol/sdk`를 import할 때, 평탄화된 구조에서 `_zod@3.25.76` 변형이 노출되면 해당 헬퍼는 내부적으로 `zod@3.x` API를 사용하게 됩니다. 반면 앱의 나머지 코드는 `zod@4.4.3`을 사용합니다. 이 타입/런타임 불일치는 **빌드 시점에는 아무 경고 없이** 발생합니다 — 양쪽 다 `@modelcontextprotocol/sdk@1.29.0`으로 해석되기 때문입니다.

`pnpm-lock.yaml:5593-5599`에서 `zod-to-json-schema@3.25.2`도 동일한 패턴으로 `zod@3.25.76`와 `zod@4.4.3` 두 변형으로 분리되어 있는 것으로 보아, 이 위험은 이 워크스페이스의 실제 의존성 그래프에 내재해 있습니다.
