## 핵심 해석 경로: pnpm 호이스팅 + Node.js 부모 디렉터리 순회

### 해석이 작동하는 이유

pnpm의 `.modules.yaml`에 따르면 이 워크스페이스는 다음 설정을 사용합니다:

| 설정 | 값 |
|---|---|
| `nodeLinker` | `isolated` |
| `hoistPattern` | `["*"]` |

`hoistPattern: ["*"]`은 **모든 트리 의존성을 루트 `node_modules`에 호이스팅**합니다. 따라서:

1. `apps/scout/dist/index.js` (생성된 시작 파일)가 `apps/scout/src/`의 모듈을 import하면
2. 그 모듈이 `smol-toml`이나 `zod` 같은 패키지를 import할 때
3. Node.js는 **가장 가까운 `node_modules`부터 부모 방향으로 순회**하면서 패키지를 찾습니다
4. `apps/scout/node_modules/smol-toml` → `apps/node_modules/smol-toml` → `node_modules/smol-toml` 순서로 탐색합니다

`isolated` 모드이지만 `hoistPattern: ["*"]` 덕분에 루트 `node_modules/`에 패키지가 물리적으로 존재하므로, **해당 앱의 `package.json`에 선언되지 않은 패키지라도 루트에서 해석됩니다**.

### 인접 소스 트리의 헬퍼 모듈이 작동하는 이유

이 워크스페이스에서 `packages/typescript-config`는 `@repo/typescript-config`라는 워크스페이스 패키지로, `workspace:*` 선언으로 앱들에 연결됩니다. `pnpm-workspace.yaml`:

```yaml
packages:
  - "apps/*"
  - "packages/*"
```

`packages/typescript-config`自身的 `package.json`에는 `dependencies`가 없지만, TypeScript 설정 파일(`node.json`, `base.json`)을 제공합니다. 앱의 `tsconfig.json`이 `@repo/typescript-config/node.json`을 `extends`할 때, TypeScript 컴파일러는 이 워크스페이스 패키지를 `apps/*/node_modules/@repo/typescript-config`의 symlink를 통해 해석합니다.

### 조용히 잘못된 버전에 바인딩되는 경우

**pnpm lockfile의 핵심 관찰**: `@modelcontextprotocol/sdk@1.29.0`은 피어 의존성 `zod: ^3.25.0 || ^4.0.0`을 선언합니다. 이로 인해:

```
apps/scout/node_modules/zod → zod@4.4.3 (앱이 선언한 버전)
apps/mcp-server/node_modules/zod → zod@4.4.3 (앱이 선언한 버전)
```

하지만 `@modelcontextprotocol/inspector` (devDependency)는 `zod: ^3.25.76`를 요구합니다. pnpm은 이를 **별도 디렉터리에 격리 설치**합니다:

```
.zpn/@modelcontextprotocol+sdk@1.29.0_zod@3.25.76/  ← inspector용
.zpn/@modelcontextprotocol+sdk@1.29.0_zod@4.4.3/    ← 앱 코드용
```

**잘못된 버전 바인딩 시나리오**:

1. **`skipLibCheck: true`가 타입 충돌을 숨깁니다** — `packages/typescript-config/base.json`의 TypeScript 설정이 `skipLibCheck: true`를 사용하므로, zod@3과 zod@4 간의 타입 불일치가 빌드 타임에 감지되지 않습니다.

2. **런타임 `instanceof` 충돌** — zod@3과 zod@4는 호환되지 않는 마이저 버전입니다. 만약 코드에서 `value instanceof zod.ZodType` 같은 검사를 하고, 런타임에 예상치 못한 zod 버전이 해석되면 조용히 실패합니다.

3. **릴리스 패키징 시 의존성 누락** — `pnpm`의 `--prod` 플래그 없이 `tsc`만으로 빌드하면 `dist/` 폴더에 JavaScript만 생성됩니다. `node_modules`의 패키지들은 `dist/`에 포함되지 않습니다. 릴리스 배포 시 `npm install --omit=dev` 또는 번들러 없이 배포하면, 앱의 `package.json`에 선언된 의존성만 설치됩니다. 헬퍼 모듈이 **선언되지 않은 패키지를 루트 호이스팅에 의존**하고 있었다면, 릴리스 환경에서 모듈을 찾지 못해 `MODULE_NOT_FOUND`가 발생합니다.

### 요약

| 항목 | 내용 |
|---|---|
| **작동 경로** | pnpm `hoistPattern: ["*"]` → 루트 `node_modules` 호이스팅 → Node.js 부모 순회가 인접 패키지를 찾음 |
| **잘못된 바인딩 조건** | 1) `skipLibCheck: true`로 타입 충돌 무시, 2) 피어 의존성 범위가 겹치는 패키지(zod v3/v4), 3) 릴리스 시 `--omit=dev` 또는 번들링 없이 배포 |
