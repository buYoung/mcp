이 리포지토리에는 Angular 코드가 없지만, 질문은 Angular 프레임워크의 `SelectControlValueAccessor`와 `Option` 지시자 내부 동작에 관한 것이므로 프레임워크 소스 기준으로 답변드리겠습니다.

---

### 모델 재적용 시점: `writeValue()`

`SelectControlValueAccessor._writeValue()`는 **Forms 계층(`FormControl`/`NgModel`)이 모델 값을 변경할 때만** 호출됩니다. 자식 `Option`이 추가·제거·변경될 때 부모 select에 어떤 콜백도 발생하지 않습니다. `_writeValue()` 내부 동작은:

```
for each registered option in _optionMap:
    if _compareWith(optionValue, modelValue):
        option.element.selected = true
    else:
        option.element.selected = false
```

즉, **"모델 자체가 바뀔 때"만 전체 옵션 순회가 일어납니다.**

### 자식 변이가 즉시 재선택을 트리거하지 않는 이유

`Option` 컴포넌트의 `ngDoCheck()`가 **자기 자신만** 평가합니다:

```
// Option.ngDoCheck()
const isSelected = parentSelect._compareWith(this.value, parentSelect._value);
// → self.selected = isSelected 로 독립 적용
```

이렇게 설계된 이유는 세 가지 불변식을 만족하기 위해서입니다:

1. **비교 폭발 방지**: `_compareWith`는 옵션 1개당 O(1) 비교입니다. 옵션 N개 추가/제거 시 전체 O(N)으로 끝나고, 각 변이마다 O(N) 순회를 유발하지 않습니다. O(N²) 복잡도를 차단합니다.

2. **DOM 드리프트 방지**: `<select>` 요소의 `selectedIndex`를 건드리지 않고 각 `<option>`의 `selected` 속성만 독립 설정합니다. 사라지는 option이 DOM에 잠시 남아있어도, `_value`(폼 모델)은 변경되지 않으므로 다음 change detection 사이클에서 남은 옵션들이 각자 정확한 `selected` 상태를 확보합니다. 브라우저가 임의로 다른 option을 선택하도록 유도하는 코드가 없습니다.

3. **일괄 처리**: Angular의 change detection은 컴포넌트 단위로 배칭됩니다.同一个 틱 내에서 N개 option이 추가/제거되어도 `_writeValue()`는 한 번도 호출되지 않으므로, 중간 상태에서 잘못된 option이 잠깐이라도 선택되는 일이 없습니다.

### 요약

| 질문 | 답변 |
|------|------|
| 모델 재적용 규칙 | `writeValue()` — FormControl/NgModel 값이 프로그래밍적으로 바뀔 때만 |
| 자식 변이 왜 차단 | 각 Option이 `ngDoCheck()`에서 자기 자신만 `_compareWith(self, parent._value)`로 평가하고, 부모의 전체 순회(`_writeValue`)를 유발하지 않음 |
| 근본 목적 | 비교 O(N) 유지 + `_value` 불변으로 브라우저 드리프트 미연에 방지 |
