이 질문의 핵심은 "미사용으로 보이는 폴백 필드들이 실제로 어떻게 관찰 가능한지"입니다. 두 가지 구체적인 메커니즘이 있습니다.

## 1. `symbol_fallback` — 폴백 필드가 쿼리 실패 여부를 바꾸는 이유

`SearchResult.symbol_fallback` (`engine.rs:32`)은 단순한 부울 플래그가 아닙니다. `search` 도구의 렌더링 분기점입니다:

**`ranking.rs:601-607`** — 매칭된 심볼이 비어 있으면 `symbol_fallback = true`로 설정되고, 모든 심볼이 폴백으로 채워집니다:

```rust
let symbol_fallback = matched_symbols.is_empty();
if symbol_fallback {
    matched_symbols = all_symbols;
}
```

**`search/mod.rs:505`** — 이 플래그가 `true`이면 상세 스니펫 대신 이름-리스트로 렌더링됩니다:

```rust
if res.symbol_fallback {
    // Symbol names + ranges ONLY (no snippets) — never `cat` the file
```

즉, 엔지니어가 `symbol_fallback = true`로 센티넬 값을 넣으면, 해당 결과는 **스니펫 없이 이름만 표시**됩니다. 이것이 "쿼리를 던지느냐 마느냐"를 결정하는 것입니다 — 빈 파일 헤더(Child 03)를 방지하기 위해 `all_symbols`로 폴백하는 로직이므로, 이 필드를 잘못 설정하면 의미 없는 심볼 목록이 표시되거나 에이전트가 엉뚱한 파일을 탐색하게 됩니다.

## 2. `path` 단일 필드 — 하나의 값이 세 가지 타입을 전환

**`overview.rs:17-20, 50-60`** — `path` 파라미터 하나의 값에 따라 완전히 다른 뷰 타입이 반환됩니다:

```rust
let path = ["path", "file_path", "file", "query"]
    .iter()
    .find_map(|key| ctx.arguments.get(*key).and_then(|v| v.as_str()))
    .filter(|p| !p.is_empty() && *p != ".");
// ...
let codemap_text = if let Some(p) = path {
    // 폴더 → FolderCodemap, 파일 → DetailsCodemap
```

- 값 없음/빈 값 → `RootCodemap` (전체 리포 구조)
- 폴더 경로 → `FolderCodemap` (파일/심볼 목록)
- 파일 경로 → `DetailsCodemap` (심볼 라인 범위)

`query` alias가 `search` 도구의 파라미터 이름과 동일하지만, `overview` 호출 시에는 `path`로 해석됩니다. 엔지니어가 `query: ""`로 빈 문자열을 넣으면 `path` 필터에서 탈락하여 **루트 뷰**가 반환되고, `query: "src/foo.ts"`를 넣으면 **파일 뷰**가 됩니다. 같은 필드의 센티넬 값이 반환 타입을 완전히 바꿉니다.

## 요약

이 코드베이스에서 폴백 필드가 관찰 가능한 이유는 두 가지 설계 원칙 때문입니다:

1. **`symbol_fallback`**은 렌더링 경로 자체를 전환합니다 — 스니펫 vs 이름-리스트, 그리고 빈 헤더 방지 로직이 이 플래그에 의존하므로 "읽히지 않는" 필드가 사실은 **결과의 구조를 결정**합니다.

2. **`overview`의 `path`**는 단일 필드가 `CodemapView`의 세 가지 구현체(`RootCodemap`/`FolderCodemap`/`DetailsCodemap`) 중 하나를 선택하므로, 어떤 타입이 반환되는지에 따라 에이전트의 후속 호출 패턴이 달라집니다.

즉, "아무도 읽지 않는 필드"라는 가정이 틀린 이유는 이 필드들이 **라우팅/분기 조건**으로 사용되기 때문입니다 — 단순한 데이터 슬롯이 아닙니다.
