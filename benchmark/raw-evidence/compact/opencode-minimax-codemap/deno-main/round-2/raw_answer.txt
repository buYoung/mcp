## Resolution Path

The packaging flow works through **pnpm workspace dependency hoisting**. In this monorepo:

- `apps/mcp-server` declares `zod: ^4.4.3` and `smol-toml: ^1.6.1`
- `apps/scout` also declares `zod: ^4.4.3` and `smol-toml: ^1.6.1`
- The root `package.json` has no direct dependencies for these

When building, pnpm's resolution traverses up the workspace hierarchy. If a helper module from a neighboring source tree (e.g., `apps/scout/src/tools/`) imports `zod` without declaring it in `apps/scout/package.json`, Node.js will still resolve `zod` because it's available in the combined workspace dependency graph—either via the importing package's own `node_modules` or via hoisting to a shared location.

## When It Binds the Wrong Version

The problem occurs when **different packages in the workspace declare different versions** of the same dependency:

```json
// apps/mcp-server/package.json
"zod": "^4.4.3"  // resolves to 4.4.3

// apps/scout/package.json  
"zod": "^4.0.0"   // resolves to 4.0.0
```

If a helper module in `apps/scout` imports `zod` but only `apps/mcp-server`'s `zod@4.4.3` is installed at runtime (due to installation order or hoisting decisions), the module silently binds to **4.4.3** even though `apps/scout/package.json` declared `^4.0.0`. This can cause runtime failures if APIs differ between versions.

pnpm's strict mode (`hoist-pattern = []`) would catch this at install time, but the default behavior allows the mismatch to pass until build/runtime.
