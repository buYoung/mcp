## Resolution Path

The packaging flow works through **pnpm workspace hoisting**. The `node_modules/.pnpm-workspace-state-v1.json` shows:

```json
"hoistPattern": ["*"],
"hoistWorkspacePackages": true
```

When a helper module (e.g., from `apps/scout` or `apps/mcp-server`) imports a package that isn't in its own `package.json`, pnpm resolves it via the **workspace root's `node_modules`** - the package was brought in by the overall release/app that depends on it.

## Quiet Version Binding Risk

**When two packages need different versions of the same dependency**, pnpm hoists the "winning" version (typically root's or first-declared) and all packages silently use that version. For example:

- Root/Release app declares `x@1.0`
- Helper module actually needs `x@2.0` (but didn't declare it properly since it worked via root's hoist)
- At runtime, helper silently gets `x@1.0`, causing subtle behavioral bugs

The `allowBuilds: { esbuild: true }` setting suggests esbuild may be involved in creating the distributable, which would bundle based on what Node resolves at build time - potentially capturing the wrong hoisted version.
