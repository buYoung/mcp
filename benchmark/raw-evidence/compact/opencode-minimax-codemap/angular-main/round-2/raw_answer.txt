The angular-main in `benchmark-data/` is the Angular framework repository itself — your question is about **Angular's own change-detection contract for object-bound form fields**.

## The Two Rules

### 1. Reference-equality gate for model→widget re-application

Angular Forms does **no deep equality check** on `setValue()` or `ngModel` binding. The single rule is:

```
if (newReference !== oldReference) → re-apply to widget
```

In `AbstractControl.setValue()` (packages/forms), the value is unconditionally written to `_value` and the lifecycle proceeds. The widget only gets told to re-select when the **parent component passes a new object reference** into the bound property — a mutated existing object will not propagate.

### 2. `ngFor` `trackBy` / shallow diffing prevents child mutations from cascading

In `NgForOf` (`packages/common/src/directives/ng_for_of.ts`), the per-item identity decision is delegated to `trackBy`:

```typescript
// No trackBy supplied → default is reference equality on the item itself
this.trackByFn = trackByFn ?? ((index: number, item: any) => item);
```

During change detection, Angular calls `trackBy(index, item)` for each item and **only** re-creates DOM nodes when the returned identity changes. This means:

- A child's **internal property mutation** (`obj.name = 'new'`) — the array reference is unchanged, `ngFor` sees the same `item` reference per trackBy, no DOM touch
- An **item being added/removed/replaced** — array reference or trackBy identity changed → Angular reconciles the list

This is **intentional**: Angular stops at the first level of the object graph (shallow change detection). If mutations deep inside a bound object automatically triggered full re-selection, every keystroke in a nested property would cause dropdown re-renders, equality-hash thrashing, and cursor/selection loss on `<select>` elements.

### Why both invariants hold simultaneously

| Concern | Mechanism |
|---|---|
| Equality checks don't balloon | No deep comparison — only `!==` on the reference; `trackBy` identity is O(1) |
| Disappearing item doesn't drift selection | `trackBy` identity change forces immediate DOM removal; `setValue()` on the control sees the new reference and updates the widget's selected index synchronously |

The design is that Angular **opt-in to deep reactivity** via `OnPush` + observables, not by default — form fields are explicitly protected from cascade invalidation because shallow change detection terminates the diff at the binding boundary.
