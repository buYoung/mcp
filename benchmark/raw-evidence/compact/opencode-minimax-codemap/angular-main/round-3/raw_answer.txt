Angular uses `afterNextRender` + a `_queuedWrite` flag to batch all child mutations into a single `writeValue()` call after the DOM settles.

**The rule** (`select_control_value_accessor.ts:173-192`):
Every option add/change/destroy calls `_writeValueAfterRender()`, but the `_queuedWrite` boolean ensures `writeValue(value)` fires **once** in the next render phase — not per child mutation.

**Why individual mutations are suppressed** (three reasons in the comments at lines 154-169):

1. **Exponential `compareWith` cost** — calling `writeValue` per option would make `_compareValue` calls grow as O(n²) when rebuilding an option list (issue #41330)
2. **Safari/IE11 DOM ordering bug** — Angular would set the select's value before the option's text node was appended, causing wrong initial selection (issue #14505)
3. **Animation removal timing** — when animations delay element removal, Angular would change the select value *before* the old option actually leaves the DOM; the browser would then auto-select the first remaining option even though it doesn't match the model (issue #18430)

The `_queuedWrite` guard collapses N concurrent option mutations into a single post-render re-evaluation, keeping equality checks linear and preventing spurious intermediate selections.
