The fallback fields remain observable because they **flow through the `CandidateFile` → `SearchResult` chain and control derived fields** that change query behavior:

1. **Fallback creates sentinel `ExtractedFile`** (ranking.rs:322-328) with empty `symbols: Vec::new()`, `literals: Vec::new()`

2. **These empty vectors determine `symbol_fallback`** (ranking.rs:604):
   ```rust
   let symbol_fallback = matched_symbols.is_empty();
   if symbol_fallback {
       matched_symbols = all_symbols;
   }
   ```

3. **`symbol_fallback` skips annotation entirely** (annotate.rs:412-415, 431-434):
   ```rust
   if req.is_fallback {
       continue; // caller scan skipped
   }
   ```

4. **The "throw" vs "not throw"** — without `unwrap_or_else`, `serde_json::from_str` returning `Err` would propagate via `?` and fail the entire search batch. With the fallback, malformed JSON silently succeeds but sets `symbol_fallback = true`, producing an empty result instead of an error.

5. **One-field `owner: Option<String>` surfaces different types** — `#[serde(default)]` on `ExtractedSymbol.owner` means missing field → `None` (not an error), but present field → `Some(String)`. The downstream `SearchRankingSignal.symbol_owner: Option<String>` exposes this schema-version difference to the plan.

The discarded sentinel values (`Vec::new()`) are never truly discarded — they set `symbol_fallback = true`, which **silently suppresses the entire annotation pass** (caller/callee analysis), changing the query from "annotations rendered" to "no annotation data".
