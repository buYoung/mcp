## 1. 미사용 슬롯이 여전히 던지기를 변경하는 이유

`dictGetOrDefault`의 멀티-속성(Tuple) 형태는 기본값 Tuple 전체를 `castColumnAccurate`로 캐스팅합니다. 선택되지 않은 슬롯의 타입이 불일치하면 해당 슬롯에서 `TYPE_MISMATCH` 예외가 발생합니다.

`src/Analyzer/Passes/DictGetTupleElementPass.cpp:164-169`의 주석이 이것을 명시적으로 설명합니다:

> `dictGetOrDefault` casts the entire default tuple to the dict's attribute tuple type at execution (see `castColumnAccurate` in `FunctionDictGetNoType::executeImpl`). If the element types of the default tuple don't match the dict's attribute types, **that cast can throw on an un-selected element** (for example, a string default for a `UInt64` attribute).

즉, 쿼리가 `dictGet(dict, key, (a, b, c))` 형태로 멀티-속성을 요청할 때, `c`를 읽지 않더라도 `c`의 기본값이 `UInt64` 속성에 대해 `String`이면 캐스팅에서 예외가 발생합니다. 이는 `src/Dictionaries/DictionaryHelpers.h:386-392`의 `DictionaryDefaultValueExtractor` 생성자에서 `checkAndGetColumn`이 실패하면 `TYPE_MISMATCH`를 던지는 것과 같은 메커니즘입니다.

## 2. 싱글 필드 형태가 다른 타입을 표면화하는 이유

`dictGet`의 싱글-속성 형태는 키 인자로부터 `LowCardinality` 래퍼를 보존하지만, 멀티-속성(Tuple) 형태는 이를 제거합니다 — Tuple은 `LowCardinality`가 될 수 없기 때문입니다.

`src/Analyzer/Passes/DictGetTupleElementPass.cpp:246-251`:

> The single-attribute form of `dictGet` preserves a `LowCardinality` wrapper from the key argument, while the tuple-attribute form drops it (a tuple cannot be `LowCardinality`). **If the result types differ, the rewrite would change the type observed by the parent node** and break the surrounding query tree.

`src/Functions/FunctionsExternalDictionaries.h:404-414`의 `getReturnTypeImpl`도 이 분기를 명시합니다:

```cpp
if (attribute_types.size() > 1)

if (key_is_nullable)

return attribute_types.front();
```

멀티-속성이면 `DataTypeTuple`을 반환하고, 싱글-속성이고 키가 nullable이면 `makeNullable()`로 감쌉니다. 같은 속성이라도 형태에 따라 반환 타입이 달라집니다.

## 요약

미사용 슬롯이 "버려져도 되는" 것이 아닌 이유:

- **던지기 관측 가능성**: 기본값 Tuple의 전체 요소에 대해 `castColumnAccurate`가 실행되므로, 읽지 않는 슬롯의 타입 불일치도 예외를 발생시킵니다.
- **타입 관측 가능성**: 싱글/멀티 형태는 `LowCardinality` 포장 여부와 `DataTypeTuple` 여부가 달라져, 상위 쿼리 플랜이 관찰하는 타입이 변경됩니다.

`DictGetTupleElementPass`는 이 두 가지 위험을 모두 인지하고, 타입이 다를 때는 최적화를 중단합니다 (`line 250`).
