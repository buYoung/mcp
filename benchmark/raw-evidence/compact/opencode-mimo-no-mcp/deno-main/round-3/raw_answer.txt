이 코드베이스에서 해당 동작을 설명하겠습니다.

## 핵심 해석 경로

이 모노레포는 **번들러 없이 순수 `tsc`**로 빌드합니다. 런타임에 모든 `import`를 Node.js가 직접 `node_modules`에서 해석합니다.

pnpm의 **strict hoisting** 기본 동작:

1. 각 패키지는 자기 `node_modules/`에 선언된 의존성만 가집니다
2.但在 워크스페이스 루트의 `node_modules/`에는 **모든 워크스페이스 패키지의 의존성이 함께 hoist** 됩니다
3. Node.js 모듈 해석은 현재 모듈 위치에서 시작해 **상위 `node_modules/`를 순회**합니다

즉, 보조 모듈이 인접 소스 트리에서 가져와졌고, 그 모듈의 `package.json`에 선언되지 않은 패키지를 import 하더라도:
- 빌드 타임: `tsc`는 타입만 체크하므로 **선언 여부와 무관하게 컴파일 성공**
- 런타임: Node.js가 상위 디렉토리를 순회하면서 **루트 `node_modules/`에서 해당 패키지를 찾음**

## `createRequire` 패턴 — 이 코드베이스의 핵심 사례

`apps/mcp-server/src/agents/common/local-binary.ts:2,10`:

```typescript
const requireFromCurrentModule = createRequire(import.meta.url);
const packageJsonPath = requireFromCurrentModule.resolve(`${packageName}/package.json`);
```

이 패턴은 **`import.meta.url`을 기준으로 resolution 컨텍스트를 생성**합니다. 현재는 문제가 없지만(선언된 의존성이므로), 만약 이 모듈이 **번들된 단일 파일로 패키징**되면:

- `import.meta.url`이 **번들 파일의 위치**를 가리킴
- `createRequire`가 번들 위치 기준으로 패키지를 탐색
- 루트 `node_modules/`에서 **hoist된 버전**을 바인딩

## 조용히 잘못된 버전을 바인딩하는 경우

동일한 패키지에 대해 두 워크스페이스 패키지가 **서로 다른 버전**을 선언할 때:

```
apps/scout/package.json:     "zod": "^4.4.3"
apps/mcp-server/package.json: "zod": "^4.4.3"    (같은 버전이면 문제 없음)

하지만 만약:
apps/scout/package.json:     "some-lib": "^2.0.0"
apps/mcp-server/package.json: "some-lib": "^3.0.0"
```

pnpm strict 모드에서는 두 버전을 **별도 위치에 설치**하지만, 번들러가 루트 컨텍스트에서 해석하면 **하나의 버전만 선택**됩니다. 보조 모듈이 기대한 버전과 다른 버전이 바인딩되어도 런타임 오류가 나지 않는 한 (API가 호환되는 경우) 조용히 다른 동작을 수행합니다.

이 코드베이스에서는 현재 번들러가 없어 이 문제가 드러나지 않지만, 향후 `tsup`/`esbuild` 등으로 단일 파일 배포를 도입하면 정확히 이 지점에서 phantom dependency 문제가 발생합니다.
