## Why Discarded Fallback Fields Remain Observable

**The core problem**: In `apps/codemap-search/src/parser/types.rs:36-37`:

```rust
#[serde(default, skip_serializing_if = "Option::is_none")]
pub owner: Option<String>,
```

The `#[serde(default)]` attribute means missing `owner` fields deserialize to `None` rather than failing. But this creates a subtle trap:

1. **Missing `owner`** → `None` → `term_hits_owner()` at `ranking.rs:106-112` returns `false` (no owner matching)
2. **Explicit empty `owner: ""`** → `Some("")` → same `false` result from `term_hits_owner()`
3. **Any other string `owner`** → `Some(string)` → WILL participate in owner matching!

Engineers stuff sentinel junk (empty strings) thinking `owner` is never read. But the **type itself** (`Option<String>`) surfaces a different shape to serde, and more importantly, `has_owner_exact_symbol()` at `ranking.rs:339-344` will include/exclude candidates based on whether `owner` is `None` vs `Some`.

**The one-field form issue**: When only `name` is populated in JSON but `owner` is absent, serde defaults it to `None`. This `None` propagates through `symbol_fallback` logic at `ranking.rs:604-606` — if no symbols match via owner, the system falls back to `all_symbols`, changing snippet rendering behavior even though the caller only reads the matched symbols list.

**Why this matters for query throws**: At `ranking.rs:321-328`, if `extracted_json` is malformed, `serde_json::from_str` throws rather than gracefully degrading. But even "valid" JSON with unexpected field shapes can cause type coercion failures elsewhere in the plan.
