## The re-selection rule: one deferred write per render cycle

Every child `NgSelectOption` mutation — setting `ngValue`, setting `value`, or being destroyed — calls `_writeValueAfterRender()` on the parent `SelectControlValueAccessor` instead of calling `writeValue()` directly.

`_writeValueAfterRender` (`select_control_value_accessor.ts:173`) applies a boolean gate and then schedules the real work:

```typescript
_writeValueAfterRender(): void {
  if (this._queuedWrite || this.appRefInjector.destroyed) {
    return;                          // ← subsequent mutations are no-ops
  }
  this._queuedWrite = true;
  afterNextRender(
    { write: () => {
        this._queuedWrite = false;
        this.writeValue(this.value); // ← fires once, after the whole render pass
    }},
    { injector: this.appRefInjector },
  );
}
```

The rule is therefore: **`writeValue` runs exactly once per render cycle, in the `write` phase of `afterNextRender`, regardless of how many options mutated during that cycle.**

`writeValue` itself (`line 198`) then does a single linear scan of `_optionMap` via `_getOptionId`, calling `_compareWith` (defaulting to `Object.is`) once per registered option to find the match, then stamps the encoded value-string onto the `<select>` element.

## Why individual child mutations must not trigger an immediate re-selection pass

The comment block at `lines 153–170` documents three distinct bugs that make the deferral mandatory:

### 1. Exponential comparisons (issue #41330)

`_getOptionId` iterates all N entries in `_optionMap`. If `writeValue` were called synchronously on every `ngValue` setter or `ngOnDestroy`, each of N option mutations would trigger a full N-length scan, producing O(N²) `_compareWith` invocations. When options are bound to whole objects — where `compareWith` may do deep structural work — this balloons quickly. The `_queuedWrite` guard collapses all mutations in a cycle into a single O(N) pass.

### 2. Partial DOM on Safari/IE11 (issue #14505)

Angular sets the `<select>` element's `value` property before the option's child text node has been appended to the DOM. The browser cannot match an option whose text node doesn't yet exist, so it fails to deselect the first option. Deferring to `afterNextRender` ensures all child text nodes are in place before the value is stamped.

### 3. Animation-delayed physical removal (issue #18430)

When the animations module removes an option, the option directive's `ngOnDestroy` fires — removing it from `_optionMap` — but the `<option>` element remains physically in the DOM until the animation completes. If Angular re-applied the model immediately after the logical removal, the `<select>` value might be set to "no match." Then, when the animation finally detaches the element, the browser notices a selected option is gone and autonomously snaps the control to the first remaining option, silently overriding the model. Deferring until after the render write phase means the physical DOM and the logical map are already reconciled before the model is stamped.

---

## Contrast with the multi-select variant

`SelectMultipleControlValueAccessor` (`select_multiple_control_value_accessor.ts:127`) has no `_queuedWrite` guard and calls `writeValue` synchronously from each child option. This is safe there because the multi-select `writeValue` iterates options to set their `selected` property directly — the browser cannot unilaterally change a multi-select to a different selection — so the deselection-on-removal race does not apply.
