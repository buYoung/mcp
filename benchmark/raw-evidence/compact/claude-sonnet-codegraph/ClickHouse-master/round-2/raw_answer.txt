There are two distinct mechanisms, both rooted in the same underlying cause: ClickHouse function evaluation is **not semantically lazy** — a function commits to its full argument structure both at type-resolution time and at execution time, with no awareness of which outputs the caller actually consumes.

---

## Mechanism 1 — The fallback tuple is validated in full before any element is read

When you call `dictGetOrDefault('d', ['a', 'b', 'c'], key, (f_a, f_b, f_c))` and downstream only reads `.a`, the fallback argument `(f_a, f_b, f_c)` is still fully processed.

In `executeImpl` (`FunctionsExternalDictionaries.h:483`):

```cpp
auto result = castColumnAccurate(column_to_cast, result_type);
```

`result_type` here is `Tuple(T_a, T_b, T_c)` — the **full tuple type** for all requested attributes. `castColumnAccurate` must succeed on every element of that tuple before execution proceeds. Only after that succeeds is the tuple split into per-attribute default columns:

```cpp
default_cols = tuple_column->getColumnsCopy();   // line 499
```

There is no per-slot, consumer-driven shortcut. If `f_b` cannot be accurately cast to `T_b`, this line never runs — the cast at 483 throws first.

A second enforcement point is inside each per-attribute lookup, in `DictionaryDefaultValueExtractor` (`DictionaryHelpers.h:386–392`):

```cpp
if (const auto * const default_col = checkAndGetColumn<DefaultColumnType>(default_values_column_ptr))
    default_values_column = default_col;
else
    throw Exception(ErrorCodes::TYPE_MISMATCH, "Type of default column is not the same as dictionary attribute type.");
```

This checks the concrete C++ column type, not just logical type equivalence. Even a fallback slot whose result is never read by the caller passes through this check.

---

## Mechanism 2 — The function variant determines the type visible to the rest of the plan

`joinGet` and `joinGetOrNull` are structurally identical lookups but they register different return types. In `HashJoin::joinGetCheckAndGetReturnType` (`HashJoin.cpp:1191–1192`):

```cpp
if (or_null && JoinCommon::canBecomeNullable(elem.type))
    elem.type = makeNullable(elem.type);
return elem.type;
```

`joinGet` → returns `T`. `joinGetOrNull` → returns `Nullable(T)`. That return type is what `FunctionJoinGet::getResultType()` exposes to the rest of the query plan. Downstream function overload selection, NULL-propagation wrappers, and type-checking all operate against this registered type — not against the runtime data.

The exact same split exists in the `dictGet` family. In `FunctionDictGetOrNull::getReturnTypeImpl` (`FunctionsExternalDictionaries.h:1006–1016`), the `OrNull` variant wraps everything — single attribute or tuple element — in `Nullable`, while the plain `dictGet` / `dictGetOrDefault` variants return `T` directly. Switching between them for what looks like "the same one-field result" silently changes the logical type that propagates forward.

For the nullable-key case on `dictGet`, this is explicit in `getReturnTypeImpl`:

```cpp
// line 412-413
if (key_is_nullable)
    return makeNullable(attribute_types.front());
return attribute_types.front();
```

A nullable key produces `Nullable(T)` even from `dictGet`. A non-nullable key produces `T`. Same data source, same attribute, same lookup — different type in the plan.

---

## Root cause

Both effects share the same cause: **the function, not the query plan, owns the contract between arguments and outputs.** Type resolution happens at build time against the full argument list; execution validates inputs against the full result type before any data flows out. No optimizer pass prunes unused tuple elements before these checks run, so a slot that is logically dead from the consumer's perspective is still live from the function's perspective. This is why stuffing a wrong-type sentinel into a "nobody reads it" fallback slot is not safe.
