## The two mechanisms that keep "discarded" fallback fields observable

The enrichment pattern is:
```sql
tupleElement(
    dictGetOrDefault(dict, ('attr1', 'attr2'), key, (default1, default2)),
    'attr1'
)
```
Only `attr1` is consumed. Engineers assume the `default2` slot is irrelevant junk. The `DictGetTupleElementPass` optimizer tries to collapse this into `dictGet(dict, 'attr1', key, default1)` — but it must bail out in two distinct cases, each of which shows that the discarded slots are still live.

---

### 1. The cast on the entire default tuple can throw on an unselected element

`src/Analyzer/Passes/DictGetTupleElementPass.cpp:164–182`

```cpp
/// `dictGetOrDefault` casts the entire default tuple to the dict's attribute tuple type at
/// execution (see `castColumnAccurate` in `FunctionDictGetNoType::executeImpl`). If the
/// element types of the default tuple don't match the dict's attribute types, that cast
/// can throw on an un-selected element (for example, a string default for a `UInt64`
/// attribute). After rewriting to a single-attribute call only the selected element is
/// cast, so such exceptions are suppressed. Bail out unless element types match exactly
```

At runtime, `dictGetOrDefault` calls `castColumnAccurate` on the **whole** default tuple before field-level extraction happens. If `default2` is a string but `attr2` is `UInt64` in the dictionary schema, that cast throws — even though the query never reads `attr2`. The optimizer therefore checks every element pair and refuses to rewrite unless all types match exactly (`:180`):
```cpp
for (size_t i = 0; i < default_elems.size(); ++i)
    if (!default_elems[i]->equals(*result_elems[i]))
        return;
```
The fallback tuple is not evaluated field-by-field; the whole thing is cast together at the `dictGetOrDefault` boundary.

---

### 2. The single-attribute form produces a different type than the tuple-attribute form

`src/Analyzer/Passes/DictGetTupleElementPass.cpp:246–251`

```cpp
/// The single-attribute form of `dictGet` preserves a `LowCardinality` wrapper from the
/// key argument, while the tuple-attribute form drops it (a tuple cannot be `LowCardinality`).
/// If the result types differ, the rewrite would change the type observed by the parent
/// node and break the surrounding query tree. Bail out in that case.
if (!new_dict_get_function.getResultType()->equals(*tuple_element_function->getResultType()))
    return;
```

`dictGet(dict, ('attr',), key)` is a multi-attribute call: its return type is `Tuple(T)`, and a `Tuple` cannot carry a `LowCardinality` wrapper, so the wrapper is stripped before the outer `tupleElement` peels it off. The single-attribute form `dictGet(dict, 'attr', key)` returns the attribute type directly and *does* preserve `LowCardinality` from the key. So the same logical value comes out with a structurally different type depending on which overload is used. Substituting one form for the other would silently change the type seen by every node above in the query plan.

---

### Why both of these are about the "unused" fallback fields

| Slot | What engineers assume | What actually happens |
|---|---|---|
| Fallback tuple element for unread attr | "junk value, never matters" | Still cast against the dict's schema; a type mismatch throws before any field is selected |
| Multi-attr vs single-attr form | "same field, same result" | Different `LowCardinality` propagation rules produce different types for the surrounding plan |

Both effects exist because the multi-attribute `dictGetOrDefault` is a single function node whose type contract and runtime cast cover the full tuple. Field selection happens downstream (`tupleElement`), after the whole tuple has already been validated and typed. The optimizer's two `return` bailouts are precisely the guards that acknowledge these two constraints.
