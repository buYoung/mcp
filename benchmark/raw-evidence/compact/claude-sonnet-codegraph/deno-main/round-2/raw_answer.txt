## The Packaging Flow

### What's happening

`cli/tools/compile.rs:compile_inner()` runs `detect_framework()` on the source directory. For a framework like **Next.js**, `detect_nextjs()` (`cli/tools/framework.rs:118`) generates a temporary entrypoint file — the **startup file from generated output**:

```ts
// .deno_compile_entry_XXXXXXXX.ts (written to disk, compiled, then deleted)
import { nextStart } from "npm:next@^{version}/dist/cli/next-start.js";
await nextStart({ hostname: "0.0.0.0" }, import.meta.dirname);
```

The **neighboring source tree helper** is the framework's build output (`.next/`, `build/`, `.output/`, etc.), added via `include_paths`. When `nextStart` boots, it loads files like `.next/server/app/page.js`. Those files contain **plain bare `require('react')`** (not `npm:react`) — the "plain package import." The `.next/` directory has no `package.json` listing `react` as a dependency, hence "its own nearby manifest would not seem to justify" the import. Yet the build succeeds because the project root's `node_modules/react` was installed as part of the overall project.

---

### Resolution path

The bare specifier from inside the framework output goes through **Node.js-style `node_modules` ancestor traversal**, implemented in `ByonmNpmResolver::resolve_pkg_folder_from_deno_module_req()` at `libs/resolver/npm/byonm.rs:132`. The critical inner function:

```rust
// byonm.rs:137-159
fn node_resolve_dir(...) -> std::io::Result<Option<PathBuf>> {
  for ancestor in start_dir.ancestors() {   // walk up the directory tree
    let node_modules_folder = ancestor.join("node_modules");
    if ... sys.is_dir(...&node_modules_folder) {
      let sub_dir = join_package_name_to_path(...node_modules_folder, alias);
      if sys.is_dir(...&sub_dir) {
        return Ok(Some(canonicalize_path_maybe_not_exists(sys, &sub_dir)?));
      }
    }
  }
  Ok(None)
}
```

Since `.next/server/app/node_modules/react` and `.next/node_modules/react` don't exist, the walk reaches the project root and binds `node_modules/react` there — the version installed by the project's own `deno install` / `npm install`.

### When it silently binds the wrong version

**The Next.js case — major-only version pinning:**  
`detect_package_version()` (`framework.rs:364`) extracts only the **major** integer from `package.json`:

```rust
ver_str.chars()
  .skip_while(|c| !c.is_ascii_digit())
  .take_while(|c| c.is_ascii_digit())  // stops at first non-digit → "15", not "15.1.2"
  .collect::<String>().parse().ok()
```

The entrypoint becomes `npm:next@^15`. Deno resolves that at `deno compile` time to **the latest `^15` available in the registry**. But `.next/` was pre-compiled with whatever minor/patch was locally installed at `next build` time. If `15.2.0` has shipped since the build, `deno compile` picks it up for the startup half while `.next/` still contains artifacts from `15.1.2` — the two halves of the running binary silently use diverging Next.js versions.

**The general byonm/bare-specifier case — transitive hoisting:**  
For a framework like SvelteKit with `build/index.js` doing `require('polka')`, the byonm resolver (`byonm.rs:213–344`) first walks ancestor `package.json` files for a dep declaration. Neither `build/package.json` (absent) nor the root `package.json` lists `polka` directly — it's a transitive dep of `@sveltejs/adapter-deno`. The resolver then falls through to the ancestor `node_modules` walk. It accepts **whichever version npm hoisted** to the root `node_modules/`. If another package in the project also requires `polka` at a different range, and npm's resolution hoisted the other package's version, the build output silently runs with an incompatible `polka`, with no version check possible since no `package.json` anchors the constraint.
