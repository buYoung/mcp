## The deferred-write gate and why it exists

### 1. The rule that decides when the model is re-applied

Every individual `<option>` child (`NgSelectOption`) calls `_writeValueAfterRender()` on its parent `SelectControlValueAccessor` in three situations:

- when its `ngValue` input is set (line 278)
- when its `value` input is set (line 289)
- when it is destroyed, i.e. removed from the list (line 300 in `ngOnDestroy`)

`_writeValueAfterRender()` (lines 173â€“192) does **not** call `writeValue` immediately. Instead it:

1. Checks `_queuedWrite`. If it is already `true`, it **returns immediately** â€” this is the coalescing gate.
2. Otherwise sets `_queuedWrite = true` and registers a single `afterNextRender` callback in the **write phase**.
3. When that callback fires (after the entire current render cycle has flushed), it resets `_queuedWrite = false` and calls `writeValue(this.value)`.

`writeValue` (lines 198â€“206) then calls `_getOptionId(value)` (lines 225â€“229), which iterates `_optionMap` and calls `_compareWith` once per registered option until it finds a match, then sets `select.value` to the corresponding opaque key string.

**The controlling rule, in one sentence:** re-application happens exactly once per render cycle, deferred to the `afterNextRender` write phase, regardless of how many options mutated during that cycle.

### 2. Why individual mutations are intentionally blocked from triggering an immediate pass

The `_queuedWrite` flag is the intentional block. The comments at lines 153â€“171 name three distinct bugs that would reappear without it:

**Bug 1 â€” quadratic comparisons (issue #41330).**  
`_getOptionId` walks the entire `_optionMap` and calls `_compareWith` for each entry. If `writeValue` were called once per option mutation, adding or replacing N options would call `_compareWith` O(NÂ²) times â€” fatal in a large picker populated from a server response. With the gate, there is exactly one linear scan per render cycle.

**Bug 2 â€” premature DOM write before child text nodes are attached (issue #14505).**  
In Safari and IE 11, setting `select.value` before an `<option>`'s text node is appended to the DOM caused the first option to remain selected instead of being deselected when no option matched the model. Deferring to `afterNextRender` guarantees all child text nodes are in the DOM before the write.

**Bug 3 â€” browser hijacks the value during animated removal (issue #18430).**  
When the animations module removes an option, the DOM node lingers while the leave animation plays. If `writeValue` were called the moment the option's `ngOnDestroy` fires (i.e., before the animation completes), Angular would write the model value while the soon-to-be-removed option still exists. The moment that option's node is finally detached, the browser resets `select.value` to the first remaining option on its own â€” even though that option does not match the model. Deferring the write until `afterNextRender` ensures the DOM has fully settled (the animated node is gone) before Angular touches `select.value`.
