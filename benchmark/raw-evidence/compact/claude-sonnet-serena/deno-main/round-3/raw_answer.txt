## The packaging flow and its resolution path

### The actors

| Term | Concrete artifact |
|---|---|
| Single distributable | `deno compile --bundle .` output binary |
| Startup file from generated output | The temporary `.deno_compile_entry_<hex>.ts` written by framework detection (`compile.rs:139-144`) — e.g., for Next.js it is `import { nextStart } from "npm:next@^{version}/dist/cli/next-start.js"` |
| Neighboring source tree | A local project `.cjs` file that esbuild cannot inline as ESM, so it externalizes it at its real on-disk path |
| Plain package import with no nearby manifest justification | A bare `require('some-package')` inside that `.cjs` file, which has no `"dependencies"` entry in any `package.json` sitting next to it |

### How the resolution path is built during `--bundle`

**Step 1 — esbuild externalizes the `.cjs` file.**  
When esbuild bundles the generated entrypoint and encounters a CJS module it cannot fold into the ESM output, it emits the module's absolute path as a string literal inside the CJS-from-ESM wrapper, e.g.  
`var x = require("/abs/path/to/helper.cjs")`.

**Step 2 — `rewrite_absolute_bundle_paths` intercepts every such path** (`compile.rs:575-651`).  
It finds every absolute-path string literal in the bundle, verifies the file exists on disk, rewrites it to  
`__internalResolveBundlePath("relative/path/to/helper.cjs")`,  
and simultaneously records the original `PathBuf` in `referenced_abs_paths`.

**Step 3 — non-`node_modules` paths are added to the VFS include set** (`compile.rs:246-254`).  
For each `referenced_abs_path` that does **not** live inside a `node_modules` tree, the path is pushed to `compile_flags.include`. The binary writer then embeds the file in the VFS at its real cwd-relative position. That is the "neighboring source tree" file living in the VFS exactly where `__internalResolveBundlePath` will ask for it at runtime.

**Step 4 — runtime resolution.**  
`__internalResolveBundlePath(rel)` computes the absolute path at runtime by joining `rel` to `path.dirname(fileURLToPath(import.meta.url))`, i.e. the directory of the bundle file inside the VFS. The VFS always places `node_modules` at the same cwd-relative offset as at build time (guaranteed by `fill_npm_vfs`, `binary.rs:1070`). So when `helper.cjs` calls `require('some-package')`, Node's standard ancestor-walk climbs from the helper's VFS path until it reaches the embedded `node_modules` and finds the package there. The plain bare import works because the ancestor `node_modules` — put there by the overall release, not by the helper's own manifest — is reachable from the helper's VFS directory.

### What the closure builder seeds (and what it misses)

`collect_bundle_required_packages` (`native_addons.rs:113-203`) seeds the npm-embed set from:
- native-addon packages and their dependents, and
- npm packages whose on-disk folder is a **prefix-match** for one of the `referenced_abs_paths`.

A local project `.cjs` file is **not** inside any npm package folder, so it contributes no seed. The package it `require()`s ends up embedded only if it is in the dependency **closure** of some other seed (a native addon, or another referenced path that did live inside `node_modules`).

### The wrong-version case

The silent version mismatch happens when:

1. `helper.cjs` calls `require('lodash')`.
2. At build time, the ancestor-walk from the helper's directory resolves to the hoisted `./node_modules/lodash@4.17.21`.
3. `collect_bundle_required_packages` never seeds `lodash` from the helper's path (it is not inside `node_modules`).
4. `lodash` **is** reachable via the closure of another seeded package — but that other package declared a peer/optional dep on `lodash@^3`, so the embedded version is `lodash@3.10.1`.
5. At runtime, `require('lodash')` from the helper's VFS position finds the embedded `lodash@3.10.1` — a different major version — with no warning, because the ancestor-walk logic is the same whether or not the version matches what was used at build time.

In short: the build succeeds silently because the package is present in the VFS (brought in by a different seed's closure), but the **version** embedded is determined by that other seed's declared dependency graph, not by the helper's actual build-time resolution. Any divergence between those two graphs causes the wrong version to be bound at runtime.
