이 모노레포의 구조와 모듈 해석 메커니즘을 분석했습니다. 핵심 답변은 다음과 같습니다.

## 정상 작동하는 해석 경로: Node.js 상향식 디렉토리 탐색 + pnpm hoisting

`moduleResolution: "NodeNext"` + `"type": "module"` 설정에서, Node.js는 import된 패키지를 해당 파일의 `node_modules`에서 먼저 찾고, 못 찾으면 **부모 디렉토리를 재귀적으로 탐색**합니다. pnpm 모노레포에서는 `pnpm install` 시 공유 의존성이 루트 `node_modules`로 hoist됩니다.

현재 이 코드베이스에서 두 앱이 모두 `@modelcontextprotocol/sdk`를 `dependencies`에 선언하고 있으므로:

```
apps/scout/src/some-helper.ts
  import { ... } from "@modelcontextprotocol/sdk/server/index.js"
         │
         ▼  ① scout/node_modules/ → 없음 (scoped 패키지라 hoist됨)
         ▼  ② node_modules/@modelcontextprotocol/sdk/ ← 루트에 존재
         ▼  ③ 해석 성공
```

helper 모듈 자체의 `package.json`이 이 패키지를 선언하지 않더라도, **릴리스 전체가 이미 해당 패키지를 루트 `node_modules`에 가져왔으므로** 빌드(tsc)와 런타임 모두 성공합니다.这就是所谓的 phantom dependency.

---

## 조용히 잘못된 버전을 바인딩하는 경우

pnpm은 기본적으로 **strict isolation**을 적용하지만, 루트에 `shared-workspace-lockfile: true`(기본값)와 hoisting 설정이 있으면, **다른 패키지가 선언한 다른 버전**이 루트에 hoist될 수 있습니다.

구체적 시나리오:

| 단계 | 상황 |
|------|------|
| 1 | `apps/mcp-server`가 `@modelcontextprotocol/sdk@^1.29.0` 선언 |
| 2 | `apps/scout`의 helper가 같은 패키지를 import하지만 자기 `package.json`에 미선언 |
| 3 | pnpm이 `mcp-server`의 의존성을 루트 `node_modules`에 hoist → `1.30.0` 설치 |
| 4 | helper가 루트에서 `1.30.0`을 해석 → **의도치 않은 버전으로 바인딩** |
| 5 | `1.30.0`이 breaking change를 포함하면 런타임에서 조용히 실패 |

반대 순서로도 발생합니다: helper가 원래 `1.30.0`을 기대했지만, 다른 패키지의 hoisting으로 `1.29.0`이 루트에 남으면 이전 버전이 해석됩니다.

## 현재 이 코드베이스에서의 관련성

이 프로젝트는 **`tsc`만 사용하는 번들링 없는 구조**이므로:

1. **빌드 시점**: TypeScript가 `@repo/typescript-config/node.json`의 `moduleResolution: "NodeNext"`로 타입 해석. phantom dependency가 있으면 타입 체크도 통과합니다.
2. **런타임 시점**: `dist/`의 `.js` 파일이 `node_modules`를 직접 참조. 루트 hoisting이 바뀌면 다른 버전이 로드됩니다.

`scout/AGENTS.md`에 명시된 규칙 `"Copied helpers stay local: ... do not introduce cross-app imports"`가 정확히 이 위험을 겨냥한 것입니다. 의도치 않은 의존성 숨기기가 가장 빈번하게 발생하는 지점이 **워크스페이스 내에서 복사-이전된 helper 파일**이기 때문입니다.
