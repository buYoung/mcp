## 왜 사용되지 않는 폴백 필드가 여전히 관찰 가능한가

이 코드베이스의 핵심 원인은 **ClickHouse가 함수의 반환 타입을 실행 전(eagerly)에 모든 인자에 대해 추론**한다는 점입니다. `multiIf`와 `transform`이 대표적입니다.

### 1. `multiIf` — 모든 분기의 타입이 반환 타입에 기여

`src/Functions/multiIf.cpp:96-151`을 보면:

```cpp
auto for_branches = [&args](auto && f)
{
    size_t branches_end = args.size();
    for (size_t i = 1; i < branches_end; i += 2)
        f(args[i]);
    f(args.back());  // ← else 분기도 포함
};

DataTypes types_of_branches;
for_branches([&](const DataTypePtr & arg)
{
    types_of_branches.emplace_back(arg);  // 모든 분기 타입 수집
});

return getLeastSupertype(types_of_branches);  // 공통 상위 타입 계산
```

`for_branches` 람다가 **모든 then 분기와 else 분기의 타입을 수집**하고, `getLeastSupertype`가 이를 모두 포함하여 공통 상위 타입을 계산합니다. 런타임에 실제로 선택되지 않는 분기라도 타입 추론에 참여하므로, 그 분기에 incompatible한 타입을 넣으면 `NO_COMMON_TYPE` 에러가 발생합니다.

### 2. `transform` — 4번째 인자(기본값)가 반환 타입을 변경

`src/Functions/transform.cpp:101-167`에서:

- **3인자 형식** (기본값 없음): `transform(T, Array(T), Array(T)) → T`
  - 반환 타입은 입력 타입 `T`의 범위 내
- **4인자 형식** (기본값 있음): `transform(T, Array(T), Array(U), U) → U`
  - `tryGetLeastSupertype(DataTypes{type_arr_to_nested, arguments[3]})` — **4번째 인자(기본값)의 타입이 반환 타입 결정에 참여**

```cpp
// 4인자 형식 (line 157)
auto ret = tryGetLeastSupertype(DataTypes{type_arr_to_nested, arguments[3]});
```

기본값에 sentinel junk를 넣으면 `type_arr_to_nested`와의 공통 상위 타입이 달라지거나, incompatible하면 에러가 발생합니다.

### 3. 왜 Column Pruning이 이 문제를 해결하지 못하는가

ClickHouse의 쿼리 분석 파이프라인은:

1. **타입 추론** → 모든 함수의 `getReturnTypeImpl` 호출 (모든 인자 타입 사용)
2. **쿼리 플랜 구성**
3. **불필요한 컬럼 제거** (`RemoveUnusedProjectionColumnsPass`, `removeUnusedColumns`)

**타입 추론이 컬럼 프루닝보다 먼저 일어나므로**, 사용되지 않는 컬럼의 타입이 불일치하면 프루닝이 실행되기 전에 이미 에러가 발생합니다.

### 요약

| 상황 | 원인 |
|------|------|
| 미사용 분기에 incompatible 타입 → 에러 | `getLeastSupertype`가 모든 분기 타입을 포함 |
| 3인자 vs 4인자 `transform` → 반환 타입 상이 | 4번째 인자가 `getLeastSupertype`에 참여 |
| sentinel junk가 에러를 유발 | 타입 추론이 실행 전 eagerly 수행됨 |

**근본 원인**: 타입 안전성(safety)이 실행 시점이 아닌 쿼리 분석 시점에서 보장되기 때문에, "읽지 않는 필드에는 아무거나 넣어도 된다"는 가정이 이 코드베이스에서는 성립하지 않습니다.
