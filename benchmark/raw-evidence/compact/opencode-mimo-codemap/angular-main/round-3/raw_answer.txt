이 워크스페이스는 Angular 프레임워크 소스코드입니다. 질문하신 select 라디오 컨트롤의 객체 바인딩 및 재선택 로직을 살펴보겠습니다.
핵심 파일들을 자세히 살펴보겠습니다.
핵심 파일을 모두 확인했습니다. 이제 답변을 정리하겠습니다.

## 답변

핵심은 `SelectControlValueAccessor` (`packages/forms/src/directives/select_control_value_accessor.ts`)의 두 메커니즘에 있습니다.

### 1. Angular가 모델을 다시 적용하는 규칙: `_writeValueAfterRender` (배치 + 디퍼)

각 하위 `NgSelectOption`은 `_writeValueAfterRender()`를 호출하지만, **즉시 `writeValue`를 실행하지 않습니다**. 대신:

- `_queuedWrite` 플래그를 `true`로 설정하고, `afterNextRender` 콜백으로 `writeValue`를 **단 하나** 예약합니다 (line 173-192).
- 이미 예약된 상태(`_queuedWrite === true`)에서 다시 호출되면 **조용히 무시**됩니다 (line 174).

즉, 한 타임스택 동안 옵션이 100개 추가되거나 삭제되어도 `_writeValueAfterRender`가 100번 불려도, **실제 `writeValue`는 단 한 번만** 실행됩니다. 모델→위젯 재적용은 **렌더 사이클 이후 (afterNextRender)**, **배치 단위로** 발생합니다.

### 2. 개별 하위 변경이 즉시 전체 재선택 패스를 트리거하지 않는 이유

주석(line 153-169)이 세 가지 이유를 명시적으로 설명합니다:

**① 복잡도 폭발 방지 (issue #41330)**
`writeValue`가 개별 option 추가/제거마다 호출되면, `_compareWith` 기반 비교(`_getOptionId`)가 O(N)² 수준으로 증가합니다. `Object.is` 또는 커스텀 `compareWith` 함수가 옵션 전체를 순회하므로, N개 option에 대해 N번 호출 = O(N²). 이를 배치로 해결합니다.

**② Safari/IE11 DOM 순서 문제 (issue #14505)**
option 요소의 텍스트 노드가 DOM에 아직 추가되기 전에 Angular가 `<select>`의 `value` 프로퍼티를 설정하면, 옵션 매칭이 실패하여 선택이 올바르게 해제되지 않습니다.

**③ 애니메이션 모듈과의 상호작용 (issue #18430)**
`@angular/animations`에서 선택된 option이 제거될 때, 애니메이션이 DOM 제거를 지연시킵니다. 만약 Angular가 option이 DOM에 아직 남아있는 상태에서 `writeValue`를 즉시 호출하면, select의 value를 변경하고 → 브라우저가 잔존하는 option으로 값을 되돌리고 → 애니메이션이 끝나서 option이 사라지면 → 브라우저가 첫 번째 option으로 자동 선택 drift가 발생합니다. `afterNextRender`를 사용하면 **모든 option이 DOM에 완전히 렌더링된 후**에야 선택값을 동기화하므로 이 문제가 없습니다.

### 요약

| 질문 | 답 |
|---|---|
| 언제 모델을 다시 적용하는가? | 개별 option 변경 시 `afterNextRender`로 단일 배치 예약, 렌더 사이클 완료 후 단 한번 `writeValue` 실행 |
| 왜 개별 변경이 즉시 재선택을 못 트리거하는가? | O(N²) 비교 폭발, 브라우저 DOM 드리프트, 애니메이션 지연 DOM 제거와의 충돌 방지를 위해 `_queuedWrite` 플래그로 디더링(debounce) |
