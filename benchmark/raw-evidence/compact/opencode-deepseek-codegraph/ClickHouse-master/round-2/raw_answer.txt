이 코드베이스(ClickHouse)에서 버려진 fallback 필드가 여전히 관찰 가능한 이유는 다음 네 가지 메커니즘 때문입니다:

---

**1. ActionsDAG는 전체 그래프 평가 — 모든 출력 노드가 실행됨**

`ExpressionActions::execute()`가 DAG를 실행할 때, 모든 **출력 노드**의 표현식을 완전히 평가합니다. `removeUnusedColumns` 최적화로 잘리지 않은 출력 컬럼(예: `dictGetOrDefault('dict', 'attr2', key, -1)` 같은 enrichment 표현식)은 소비자가 "읽지 않더라도" 스텝 실행 시 예외를 던지면 전체 쿼리가 실패합니다.

**2. `removeUnusedColumns`는 최적화 단계에서만 실행 — 타입 추론은 계획 구성 시 이미 발생**

계획이 먼저 **구성되고**(`addStep()` → `updateOutputHeader()` → `blocksHaveEqualStructure()` 검증), 그다음 `buildQueryPipeline()` 내 `optimize()`에서 `tryRemoveUnusedColumns`가 실행됩니다 (`query_plan_remove_unused_columns` 설정에 게이트됨). Type resolution (예: `if(cond, then, else)`의 `function->build(arguments)` → `getResultType()`)은 **모든 인자**로부터 공통 타입을 결정하며, 이는 최적화 이전에 완료됩니다. fallback 값을 `-1` (Int64)에서 `'N/A'` (String)로 바꾸면 전체 IF 식의 결과 타입이 변해 하위 계획 헤더가 달라집니다.

**3. JoinStepLogical은 더미 컬럼을 조건 없이 유지**

`JoinStepLogical::removeUnusedColumns`(`src/Processors/QueryPlan/JoinStepLogical.cpp:397`)는 `join_dummy_result_name` 더미 컬럼을 **필요하지 않아도** 유지합니다:
```cpp
else
{
    /// Dummy column kept even though not required.
    required_nodes.push_back(output_node);
}
```
이 더미 컬럼의 타입(`DataTypeUInt8`, 0 리터럴)은 `absorbExtraChildColumns()`를 통해 부모의 입력 헤더로 전염되어, 부모의 함수 오버로드 해석이나 재료화 동작에 영향을 줍니다.

**4. `removeUnusedActions`는 출력에서 제거된 노드만 재귀적으로 가지치기**

`ActionsDAG::removeUnusedActions`(`src/Interpreters/ActionsDAG.cpp:729`)는 outputs에 남아 있는 노드에서 DFS로 도달 가능한 노드만 보존합니다. fallback 컬럼이 DAG outputs에 계속 남아 있다면(예: 2, 3번 이유로), 그 컬럼의 전체 표현식 트리(따라서 예외 가능성)도 함께 살아있습니다.
