## Why discarded fallback fields remain observable in `dictGet`

The mechanism lives entirely in `FunctionDictGetNoType` (`src/Functions/FunctionsExternalDictionaries.h`).

---

### Phenomenon 1 — The unused default slot can still make the query throw

When `dictGetOrDefault` is called with multiple attribute names, the entire default tuple is cast-checked eagerly, even for slots the consumer never reads.

**`getReturnTypeImpl` (line 404–409):** whenever more than one attribute name is given, the return type becomes a `DataTypeTuple` containing *all* attribute types:

```cpp
if (attribute_types.size() > 1)
    return std::make_shared<DataTypeTuple>(attribute_types, attribute_names);
```

**`executeImpl` (line 483):** the fourth argument — the whole default tuple — is immediately cast to that full `result_type`:

```cpp
auto result = castColumnAccurate(column_to_cast, result_type);
```

`castColumnAccurate` calls `createInternalCast` and executes a real `CAST` function on the *entire* tuple column (`castColumn.cpp:48–51`). There is no per-slot laziness here. If slot 2 of the default tuple contains a value whose type cannot be accurately converted to the dictionary attribute's type (e.g. a string where a UInt64 is expected), the cast throws regardless of whether any downstream expression ever reads `tupleElement(..., 2)`.

The reason sentinel junk works in isolation is that, if the sentinel value's *literal type* happens to be cast-compatible with the attribute type, the cast silently succeeds. But the moment it isn't, the exception surfaces at execution time even though no consumer touches that slot.

### Phenomenon 2 — The single-field tuple form propagates a different type to the plan

`getAttributeNamesFromColumn` (lines 781–821) accepts either a plain `const String` or a `const Tuple` of strings. Then `getReturnTypeImpl` applies this rule (lines 404–414):

```cpp
if (attribute_types.size() > 1)
    return std::make_shared<DataTypeTuple>(attribute_types, attribute_names);
// falls through for exactly one attribute:
return attribute_types.front();
```

The condition is `> 1`, not `> 0`. A call written as:

```sql
dictGet('d', ('col',), id)     -- tuple form, one name
```

produces `attribute_types.size() == 1`, so it falls through to `return attribute_types.front()` — same as the string form. That means both forms agree on the return type.

The divergence appears in `FunctionDictGetOrNull::getReturnTypeImpl` (lines 1001–1019), which wraps the base result:

```cpp
auto result_type = dictionary_get_func_impl.getReturnTypeImpl(arguments);  // line 1003
WhichDataType result_data_type(result_type);
if (result_data_type.isTuple())
{
    // makes every element Nullable
    result_type = std::make_shared<DataTypeTuple>(elements_types_copy, ...);
}
else
    result_type = makeNullable(result_type);   // single attribute path
```

Because the typed variants (`FunctionDictGetImpl`, lines 861–884) hard-code their return type to the concrete `DataType` template parameter and then re-call `impl.getReturnTypeImpl` at *execute* time to verify the dictionary attribute actually matches:

```cpp
auto return_type = impl.getReturnTypeImpl(arguments);  // line 875
if (!return_type->equals(*result_type))
    throw Exception(ErrorCodes::TYPE_MISMATCH, ...);
```

A typed call like `dictGetStringOrDefault('d', ('col',), id, tuple('fallback'))` hits the `DataTypeTuple` branch in `getReturnTypeImpl` when there are multiple attribute names, but the typed function's own `getReturnTypeImpl` returns the fixed scalar type — so the equality check at line 877 throws `TYPE_MISMATCH`. Swapping from the tuple-of-one form to the string form (or vice versa) therefore changes what type is committed to the plan, and downstream nodes that depend on the return type — type-checking in projections, `tupleElement` arity checks, or a JOIN condition comparing this result to another column — see a structurally different type.

---

### Summary

Both effects trace back to the same root: **`dictGet` builds its result type from the full set of attribute names before execution**, and **the default tuple is cast to that full result type in one atomic operation at execution time**. There is no pruning of unused tuple slots before the cast, so a "junk" default in slot N still participates in the cast validation for every row. Separately, the presence or absence of the tuple wrapper in the attribute-name argument position is the only thing that controls whether the function emits a `Tuple(T)` vs. a plain `T` into the rest of the query plan — a one-character syntactic difference (`'col'` vs. `('col',)`) that is invisible to the consumer but structurally changes the type seen by every downstream operator.
