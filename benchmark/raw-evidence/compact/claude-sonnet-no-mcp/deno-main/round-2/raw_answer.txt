## How the packaging flow keeps working

### 1. Startup file from generated output

`compile_inner` (`cli/tools/compile.rs:97-170`) detects a framework directory and writes a temporary entrypoint:

```rust
// compile.rs:139-144
let entrypoint_path = dir.join(format!(
    ".deno_compile_entry_{:08x}.ts",
    rand::thread_rng().r#gen::<u32>()
));
std::fs::write(&entrypoint_path, detection.entrypoint_code)?;
```

This generated file lives in the app directory and becomes the `compile_flags.source_file`.

### 2. Resolution of the helper's bare package specifier

When the module graph encounters `import "express"` (or any bare specifier) from the helper — which lives in a sibling/neighboring directory that isn't the entrypoint's directory — the call lands in:

**`WorkspaceResolver::resolve`** (`libs/resolver/workspace.rs:1219`)

The resolver tries three steps in order:

**Step 1 — Synthetic import map (workspace.rs:1232)**
A single import map is built by stitching together the root `deno.json`'s `imports` section with every workspace member's `imports`. It is applied to *all* referrers, regardless of whether they live inside the workspace tree. If `"express"` appears anywhere in the combined map, the specifier resolves here — even for files from a completely external neighboring directory.

**Step 3 — `entries_for_specifier` ancestor walk (workspace.rs:1386-1413)**
If the import map has no entry, the code does:

```rust
for (_dir_url, pkg_json_folder) in
    self.pkg_jsons.entries_for_specifier(referrer)        // ← key call
{
    for (bare_specifier, dep_result) in pkg_json_folder
        .deps.dependencies.iter()
        .chain(pkg_json_folder.deps.dev_dependencies.iter())
    {
        if let Some(path) = specifier.strip_prefix(bare_specifier.as_str()) ...
            return Ok(MappedResolution::PackageJson { ... });
```

`FolderScopedMap::entries_for_specifier` (`libs/resolver/collections.rs:63-105`) iterates the `BTreeMap<UrlRc, PkgJsonResolverFolderConfig>` in **reverse order** (deepest URL first) and yields every entry whose directory URL is a **prefix of the helper's referrer URL**. Because the workspace's root `package.json` is always registered in that map, and the root URL is a prefix of every workspace-relative URL, the root's dependency table is checked if no closer ancestor matched. The package `"express"` is found there, and `MappedResolution::PackageJson` is returned with the root's version range (e.g., `^4`).

**Step 4 — Managed npm resolution (npm/managed/mod.rs:216-222)**
That version range becomes a `PackageReq` fed to:

```rust
pub fn resolve_pkg_folder_from_deno_module_req(
    &self,
    req: &PackageReq,           // ← referrer is NOT a parameter here
) -> Result<PathBuf, ...> {
    let pkg_id = self.resolution.resolve_pkg_id_from_pkg_req(req)?;
    Ok(self.resolve_pkg_folder_from_pkg_id(&pkg_id)?)
}
```

The managed resolver ignores the referrer entirely. It looks up the **global npm snapshot** for the single installed version satisfying `^4` — say `4.21.1` — and returns its cache folder.

At binary write time (`cli/standalone/binary.rs:434-473`), this package folder is embedded in the VFS. The standalone runtime recreates the same `WorkspaceResolver` from the serialized `package_jsons` using `root_dir_url` as the anchor (`cli/rt/run.rs:1295-1351`), so the same resolution logic runs at execution time.

## When the wrong installed version is silently bound

There are two concrete failure modes.

**Mode A — Two workspace members, different ranges for the same package**

Suppose the workspace has:
```
/project/
  package.json           → "express": "^5"   (root)
  packages/
    webapp/
      package.json       → "express": "^4"   (webapp member)
    helper/
      src/foo.ts         → import "express"   (no package.json)
```

`entries_for_specifier` for `helper/src/foo.ts` yields ancestors in this order:
1. `packages/helper/` — no `package.json` registered
2. `packages/` — no `package.json` registered
3. `/project/` — root `package.json` has `"express": "^5"` → **match**

The helper gets `^5` from the root, never seeing `webapp`'s `^4`. If `webapp` brought `express@4.21.1` into the snapshot but the root brought `express@5.0.0`, the managed resolver picks `5.0.0` for the helper — which may be incompatible with what the helper's author tested against. No error is raised at compile time or at binary embedding time.

**Mode B — The helper is from a genuinely neighboring tree (not under the workspace root)**

If the helper lives outside `workspace_root` entirely (e.g., included via `--include` from a sibling directory), `entries_for_specifier` returns nothing (no ancestor `package.json` covers it), and step 3 finds nothing. Resolution then falls through to the import map (step 1). Whatever version range the root's `deno.json` import map names is the one used — the helper gets that version regardless of what it was written against, with no ability to express its own constraint.

In both modes the mismatch is silent because `ManagedNpmResolver::resolve_pkg_folder_from_deno_module_req` accepts only the `PackageReq` derived from the ancestor/import-map entry, not the helper's own referrer, so there is no cross-check between "what version the helper needs" and "what version the ancestor declared."
