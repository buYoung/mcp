## Why discarded fallback fields remain observable

The pattern in question is `dictGetOrDefault` with multiple attributes - the "enrichment" function. The relevant class is `FunctionDictGetNoType<DictionaryGetFunctionType::getOrDefault>` in `src/Functions/FunctionsExternalDictionaries.h`.

---

### Mechanism 1 — the unused default slot still runs through `castColumnAccurate`

When the attribute-name argument is a tuple of strings (e.g. `('attr1', 'attr2')`), `getReturnTypeImpl` returns `Tuple(T1, T2)` ([L404–409](src/Functions/FunctionsExternalDictionaries.h#L404)):

```cpp
if (attribute_types.size() > 1)
    return std::make_shared<DataTypeTuple>(attribute_types, attribute_names);
```

At execution time (`executeImpl`, [L481–483](src/Functions/FunctionsExternalDictionaries.h#L481)) the entire default-value argument — the full tuple including every "unused" slot — is cast in one shot:

```cpp
ColumnWithTypeAndName column_to_cast = {column_before_cast.column->convertToFullColumnIfConst(),
                                        column_before_cast.type, column_before_cast.name};
auto result = castColumnAccurate(column_to_cast, result_type);   // L483
```

`result_type` here is `Tuple(T1, T2)`. `castColumnAccurate` validates every element of the tuple against its corresponding attribute type. **There is no per-element opt-out.** If `default2` holds a type-incompatible sentinel (say, a String where T2 is UInt64), the cast throws — even when every downstream operator only reads `tupleElement(result, 1)` and never touches `attr2`. The unused fallback field physically changed whether the query succeeded.

---

### Mechanism 2 — the "one-field form" surfaces a scalar, not a Tuple

The return-type branch in `getReturnTypeImpl` ([L404–414](src/Functions/FunctionsExternalDictionaries.h#L404)) is guarded by `> 1`, not `>= 1`:

```cpp
if (attribute_types.size() > 1)                                   // Tuple path
    return std::make_shared<DataTypeTuple>(attribute_types, attribute_names);
...
return attribute_types.front();                                    // Scalar path
```

A call written as `dictGetOrDefault('dict', 'attr1', key, d1)` (string arg) produces return type **`T1`**.  
The "similar one-field form" `dictGetOrDefault('dict', ('attr1',), key, (d1,))` (1-element tuple arg) also produces `attribute_types.size() == 1` → also returns **`T1`**.  
But the two-attribute form returns **`Tuple(T1, T2)`**.

These are completely different types in the plan. Every downstream node — projections, predicates, aggregations, further function calls — sees a `Tuple` in one case and a raw scalar in the other. Type-checking, implicit casts, and `tupleElement` call legality all change. Collapsing from the multi-attribute form to the one-attribute form is not a neutral simplification: it replaces a `Tuple(T1, T2)` with a bare `T1` in the column header, which is observable by any operator that has already been typed against the `Tuple`.

---

### Summary

Both effects originate from the same design: `FunctionDictGetNoType` treats all requested attributes as a single unit. The **planner** always builds one `Tuple` type spanning all attributes (or a scalar for exactly one), and the **executor** always casts the entire default value as that same unified type. There is no code path that evaluates only the subset of default elements that a consumer happens to read. The "other" slots are never stripped before the cast fires, so their types are always load-bearing — both at type-resolution time (changing the plan-visible type class) and at execution time (participating in the `castColumnAccurate` that can throw).
