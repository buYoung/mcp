## The two mechanisms that keep discarded fallback fields observable

The enrichment pattern is `dictGetOrDefault(dict, ('a', 'b', 'c'), key, default_tuple)` where only one element is subsequently extracted with `tupleElement(..., 1)`. Two entirely separate paths make the other two slots matter.

---

### 1. The default-tuple cast runs on every element before extraction

**File:** `src/Functions/FunctionsExternalDictionaries.h:476-504`

At execution time, `executeImpl` does this:

```cpp
ColumnWithTypeAndName column_to_cast = { column_before_cast.column->convertToFullColumnIfConst(), ... };
auto result = castColumnAccurate(column_to_cast, result_type);   // L483
```

`result_type` here is the **full tuple type** derived from all requested attributes — e.g. `Tuple(UInt64, String, Float32)` for `('a', 'b', 'c')`. `castColumnAccurate` is called on the entire default tuple at once, before any element is separated out. If slot 2's default is a string but attribute `b` is `UInt64`, the cast throws right there — regardless of the fact that the consumer will only read slot 1. The extraction of individual default columns (`default_cols = tuple_column->getColumnsCopy()`, L499) happens *after* this cast, so a type-incompatible junk default on a slot nobody reads is still fatal.

This is the "supposedly unused fallback slot can still change whether the query throws" half of the problem.

---

### 2. The optimizer rewrite produces a different type for the one-field form

**File:** `src/Analyzer/Passes/DictGetTupleElementPass.cpp:246-251`

The `DictGetTupleElementPass` recognises the `tupleElement(dictGet(dict, tuple_of_attrs, ...), idx)` pattern and rewrites it to the cheaper single-attribute `dictGet(dict, single_attr, ...)`. But it then explicitly checks:

```cpp
/// The single-attribute form of `dictGet` preserves a `LowCardinality` wrapper from the
/// key argument, while the tuple-attribute form drops it (a tuple cannot be `LowCardinality`).
/// If the result types differ, the rewrite would change the type observed by the parent
/// node and break the surrounding query tree. Bail out in that case.
if (!new_dict_get_function.getResultType()->equals(*tuple_element_function->getResultType()))
    return;   // L251 — bail, leave the tree unchanged
```

When a dictionary attribute has type `LowCardinality(String)`, the multi-attribute (tuple) form produces `Tuple(String, ...)` — the `LowCardinality` wrapper is shed because the outer result is a `Tuple`. Extracting element 1 from that gives `String`. The single-attribute form however produces `LowCardinality(String)` directly. These are not equal, so the rewrite is suppressed. Any parent node in the plan that was typed against `String` would see `LowCardinality(String)` if the rewrite proceeded — the "discarded" attribute slots are what forced the tuple form in the first place, and their *collective* type is what suppresses the optimisation.

---

### Why the assumption fails

The slots that "nobody reads" control two independent things:

| | Mechanism | When it bites |
|---|---|---|
| **Runtime** | `castColumnAccurate` on the full default tuple (L483) | Any type-incompatible junk in any slot of the default tuple throws at execution |
| **Plan-time** | `DictGetTupleElementPass` type-equality guard (L250) | Multi-attribute tuple form drops `LowCardinality`; single-attribute form does not; rewrite is blocked, leaving the broader plan with the tuple-form's type |

In both cases the unused elements aren't dead — they are structurally part of a compound value (a tuple) that is processed or typed as a whole before any individual element is singled out.
