The answer lives entirely in `packages/forms/src/directives/select_control_value_accessor.ts`. Here is a precise reading of the two mechanics.

## 1. The rule that governs re-application of the model

### What triggers a re-selection pass

Every `NgSelectOption` calls `_select._writeValueAfterRender()` in three situations:

| Trigger | Site |
|---|---|
| `[ngValue]` input setter fires (option value set/replaced) | L278 |
| `[value]` input setter fires (simple string option) | L289 |
| `ngOnDestroy` runs (option removed from the DOM) | L300 |

### The rule itself: one deferred write per render cycle

`_writeValueAfterRender` (L173–192) is the actual gatekeeper:

```typescript
_writeValueAfterRender(): void {
  if (this._queuedWrite || this.appRefInjector.destroyed) {
    return;                          // ← short-circuit: already scheduled
  }
  this._queuedWrite = true;

  afterNextRender(
    {
      write: () => {
        if (this.destroyRef.destroyed) return;
        this._queuedWrite = false;
        this.writeValue(this.value); // ← the actual re-selection pass
      },
    },
    {injector: this.appRefInjector},
  );
}
```

The rule in one sentence: **re-apply the model exactly once, in the `write` phase of the *next* render cycle, no matter how many option mutations arrive in the current cycle.**

`afterNextRender` with the `write` phase key means execution happens after Angular has committed all pending DOM mutations for that pass. The `_queuedWrite` boolean is the de-duplicating gate — once it is `true`, every subsequent call from any `NgSelectOption` in the same cycle is a no-op.

### How the match is made (the `_compareWith` path)

Inside `writeValue` (L198–206), it calls `_getOptionId(value)` (L225–230), which **iterates `_optionMap` and calls `this._compareWith(storedOption, modelValue)` once per registered option** until it finds a match. The default comparator is `Object.is`, but the `[compareWith]` input (L133–141) lets the application supply a domain-equality function — e.g., comparing by `id` field rather than object identity. That matched entry's opaque numeric string key is then composed into a `"<id>: <value>"` string and stamped onto the `<select>` element's `value` property.

## 2. Why individual child mutations must not trigger an immediate full re-selection pass

The code documents three distinct, independently-motivated bugs that the deferred approach eliminates — each one is a separate class of failure.

### a) Quadratic `_compareWith` growth — issue #41330 (L155–156)

`_getOptionId` is O(N) in the number of registered options: it calls `_compareWith` for every entry in `_optionMap`. If `writeValue` were called immediately inside each `NgSelectOption.ngValue` setter, a list of N options would produce:

```
first option fires → 1 compare call
second option fires → 2 compare calls
...
Nth option fires → N compare calls
Total: N(N+1)/2 — quadratic in N
```

By coalescing all mutations into a single `writeValue` call at the end, the total compare-call count drops back to O(N).

### b) Browser deselects when text node is not yet in the DOM — issue #14505 (L157–162)

`<option>` elements are partially constructed: Angular sets the `value` *property* in one step and appends the *text node* child (the visible label) in another. In Safari and older IE, `<select>.value = x` was being evaluated before the option's text child was attached, which caused the browser to fail to recognize the option as selectable, leaving nothing selected even though the model matched. Waiting for `afterNextRender` guarantees the `write` phase runs after all DOM insertions are complete — the text node is already attached when `writeValue` stamps the value.

### c) Animation-module race: the browser steals the selection — issue #18430 (L163–169)

The animations module delays actual DOM removal to allow exit animations to play. The sequence that broke without deferral:

1. The selected option is logically removed (model updates, `ngOnDestroy` fires, `_optionMap` is updated).
2. `writeValue` is called immediately — sets `<select>.value` to the model's current value, which has **no matching option** (the model no longer maps to any option).
3. The option element is *still in the DOM* during its exit animation, so the browser sees the `<select>` with no selected entry and auto-promotes the first available option to selected.
4. The animation finishes, the element is removed — but the browser now owns a selection that contradicts the model.

With the deferred write, `writeValue` only executes after the next render, by which point the animation framework has already removed the element. The `<select>` DOM is in its settled, post-animation state before selection is re-applied, so the browser cannot make a spurious auto-selection.

---

## Summary

The two invariants are maintained by a single mechanism: the `_queuedWrite` boolean plus `afterNextRender({write})`.

- **Equality calls do not balloon** because all N option mutations in one cycle collapse into one `writeValue` call, keeping `_compareWith` invocations at O(N).
- **The browser cannot drift to a wrong choice** because the re-selection pass only runs after the DOM has fully settled — text nodes are attached, animations have finished removing elements — so there is no in-between state where the browser can auto-correct the `<select>` to an unintended option.
