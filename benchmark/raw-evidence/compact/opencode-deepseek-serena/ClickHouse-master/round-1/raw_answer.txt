이 질문은 codemap-search의 `engine.rs`/`ranking.rs`에 있는 **`symbol_fallback` 패턴**을 가리킵니다. 구체적으로 말하면:

1. **fallback 필드 계산의 부수효과(side effect)가 같이 읽히는 필드를 변경한다** — `ranking.rs:600-607`에서:
   ```rust
   let symbol_fallback = matched_symbols.is_empty();
   if symbol_fallback {
       matched_symbols = all_symbols;  // ← 이 줄!
   }
   ```
   `symbol_fallback`은 "아무도 읽지 않는다"고 가정하고 true/false를 채우지만, 그걸 *계산하는 과정에서* `matched_symbols`를 통째로 교체한다. 소비자가 `matched_symbols`는 읽으므로, 이 부수효과는 무시할 수 없다.

2. **Rust의 struct literal은 모든 필드를 eager 평가한다** — `ranking.rs:310-328`의 `candidate_from_doc()`를 보면:
   ```rust
   let file_path = doc.get_first(...).and_then(...).unwrap_or("");    // 소비자가 읽는 필드
   let extracted_json = doc.get_first(...).and_then(...).unwrap_or("{}"); // "안 읽힌다"고 생각
   let extracted_file: ExtractedFile = serde_json::from_str(extracted_json) // ← 여전히 실행된다
       .unwrap_or_else(|_| ExtractedFile { ... });
   ```
   `extracted_json` 파싱은 `file_path`와 무관해 보이지만, `serde_json::from_str`이 **panic하지는 않더라도** 에러 로깅, 힙 할당, `ExtractedFile` 기본값 생성을 유발한다. 만약 누군가 `unwrap_or_else` 대신 `expect()`나 `?`를 쓰면 **쿼리가 throw**한다.

3. **동일한 모양의 단일 필드 폼이 다른 타입을 표면화**한다 — 예를 들어 `symbol_fallback: bool` 필드의 존재가 `SearchResult`의 `Serialize` 출력이나 `match` 패턴 매칭에서 분기를 결정한다. 엔지니어가 "어차피 안 읽히니까 이 필드를 없애거나 타입을 바꾸자"고 하면, JSON 직렬화 스키마가 바뀌어 MCP 응답을 받는 상대방(클라이언트)이 깨진다.

요약: 언어가 **부수효과를 허용하고**(계산 중 다른 필드 변경), **모든 필드의 생성을 eager하게 실행하며**([Rust struct literal semantics](https://doc.rust-lang.org/reference/expressions/struct-expr.html)), **직렬화 스키마가 전체 타입을 기준으로 결정**되기 때문에, "아무도 안 읽는다"는 가정이 깨진다.
