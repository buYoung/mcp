# [feat] codemap-search 다단계 호출 흐름 추적

## Work Type
feat

## 배경

같은 폴더의 `agent-benchmark-followups.md` 04번 항목을 상세화한 설계 검토서다.

2026-06 벤치마크(v3→v7)에서 t4 과제 — "RPC `query` 메서드 분기 → 실제 질의 실행

다른 과제의 중앙값이 7~9턴인 것과 비교하면 3~4배 초과다. 이 브리프는 그 원인을

## As-Is: 왜 t4류가 25~33턴을 쓰는가

### 현행 callee 주석의 설계 한계

`callers.rs`의 `discover_callees` 함수(`callers.rs:407-453`)는 대상 심볼의 소스
범위를 파일에서 읽어 `identifier(` 패턴으로 직접 호출 이름을 추출한 뒤, 스냅샷의
전역 `fn` 이름 집합과 교집합을 구한다. 결과는 `callee_list_cap`(기본값 5)으로 잘린다
(`config.rs:211`). 렌더링은 `callers.rs:659`에서 `_calls (depth 1, approximate,
name-match only):_` 레이블 아래 단순 이름 목록으로 출력된다.

depth가 1로 고정되어 있으므로, t4처럼 `RpcHandler::query` → `execute_query` →
`Datastore::process` → `Transaction::run` 같은 4단계 체인은 첫 단계(`execute_query`)만
주석에 나타난다. 에이전트는 체인을 한 단계씩 전개하기 위해 다음 패턴을 반복한다:

1. 현재 심볼의 callee 목록에서 후보를 읽음 (search detail 응답)
2. 후보 심볼 정의를 확인하기 위해 `search` 또는 `read` 호출
3. 그 심볼의 callee를 얻기 위해 다시 `search` 호출
4. 이를 체인이 끝날 때까지 반복

4단계 체인이라면 최소 8~12회의 추가 도구 호출이 필연적으로 발생한다. 거기에 에이전트의
확인성 read(벤치마크에서 v7 기준 잔존 65회)가 더해지면 25~33턴이 현실적이다.

### 관련 코드 위치

| 역할 | 파일:줄 |
|---|---|
| depth-1 callee 발견 함수 | `callers.rs:407` (`discover_callees`) |
| callee 목록 렌더링 | `callers.rs:656-677` |
| callee_list_cap 설정 기본값 | `config.rs:249` (`callee_list_cap: 5`) |
| annotation_sub_budget 설정 기본값 | `config.rs:251` (`annotation_sub_budget: 8192`) |
| 주석 바이트 예산 이중-카운터 처리 | `mcp.rs:728-751` |
| 단일 워크스페이스 스캔 진입점 | `callers.rs:724` (`annotate_results`) |

### 핵심 제약

현재 워크스페이스 스캔(`scan_workspace`, `callers.rs:174`)은 caller/callee 두 방향을
combined-regex로 1회 walk한다. callee 발견은 스캔이 아니라 대상 심볼의 소스 범위를
직접 읽는 방식이라 스캔 비용을 쓰지 않는다. 이 차이가 방향 (a)와 (b)의 비용 구조를
다르게 만든다.

## To-Be: 방향 후보 평가

### (a) 디스패치형 함수에 한해 depth-2 callee 확장

**개념**: `discover_callees`가 반환한 depth-1 callee 목록 중 "디스패치형"으로 분류된
심볼에 한해 `discover_callees`를 재귀 1회 적용해 depth-2 callee를 추가로 수집한다.

**디스패치형 휴리스틱 제안**:
- depth-1 callee 수가 임계치(예: 3개 이상)인 경우 — 많은 하위 호출자를 분기하는
  패턴
- 소스 범위에 `match ` 또는 `switch ` 키워드가 존재하고 arm 수가 임계치(예: 4개
  이상)인 경우 — 명시적 분기 디스패치 패턴
- 두 조건 중 하나를 만족하면 디스패치형으로 분류

**스캔 비용 추정**: depth-2 callee 발견은 disk read(소스 범위 직접 읽기)만 사용하므로
추가 워크스페이스 walk는 없다. 디스패치형으로 분류된 심볼 수를 N이라 하면 `read_workspace_file`
호출이 N번 추가된다. t4 규모 저장소(surrealdb ~2,700 파일)에서 depth-1 callee가
5개, 그 중 디스패치형 2개라면 file read 2회 추가 — 응답 시간 영향은 무시 수준이다.

**노이즈 문제 (핵심 쟁점)**: `discover_callees`는 이름 매치 방식이므로 depth마다
귀속 오차가 누적된다. depth-1에서 이미 동명 함수가 여러 개 존재하면(`common_name_threshold`
초과) `target ambiguous` 레이블이 붙는다. depth-2는 그 위에 두 번째 이름 매치를
쌓으므로, 레이블 없이 출력되면 에이전트가 잘못된 심볼을 따라갈 위험이 있다. 정확하게
레이블링하려면 depth-2 각 항목에 `(via {depth-1-callee}, target ambiguous)` 형태의
중첩 레이블이 필요하고, 그것이 텍스트 렌더링 공간을 상당히 차지한다.

**출력 바이트 영향**: depth-1 callee 5개, 각각 depth-2 callee 3개를 생성하면 최대
15개 추가 항목. 1줄당 ~50바이트로 750바이트 증가. 현재 `annotation_sub_budget`
8192바이트(`config.rs:251`) 내에서 수용 가능하나, 디스패치형이 중첩될 경우 기존
callee/caller 항목이 예산 밖으로 밀릴 수 있다.

**결론**: 비용 자체는 낮지만, 노이즈 누적을 정직하게 레이블링하면 렌더링이 복잡해지고
에이전트가 오히려 더 많은 확인 read를 유발할 수 있다. "디스패치형" 휴리스틱의
정확도도 미검증이다.

### (b) 호출 체인 요약 응답(시작 심볼 → 끝 심볼 경로)

**개념**: 시작 심볼과 끝 심볼(또는 타입/키워드)을 받아 두 심볼 사이의 호출 경로를
탐색하고 그 체인을 한 번에 응답하는 기능.

**새 도구 vs. 기존 `search` 파라미터 확장**:
- 새 도구(`call_chain` 등): 명시적 인터페이스로 에이전트가 의도를 직접 표현할 수
  있다. 그러나 현재 5개 도구(search/overview/read/find/grep)의 분리 원칙에 도구를
  추가하는 것은 MCP 네임스페이스를 확장하는 결정이다.
- `search` 파라미터 확장(예: `chain_to` 파라미터): 기존 도구 안에 수용해 인터페이스를
  단순하게 유지한다. 단, `search`의 결과 형식이 달라져 에이전트가 두 모드를 구분해야
  한다.

**경로 탐색의 상태 공간**: 호출 그래프 탐색은 BFS/DFS로 구현하더라도 분기 수가 많은
저장소에서 지수적으로 증가한다. surrealdb 규모(~2,700 파일)라면 중간 노드가 수십~수백
개 될 수 있다. depth 상한(예: 6)과 방문 노드 수 상한(예: 200)을 함께 두어야 한다.

**응답 바이트 통제**: 체인 경로를 `A → B → C → D` 형태로 1줄 요약하면 100바이트
미만이지만, 각 노드의 파일:줄 위치와 함께 렌더링하면 노드당 ~80바이트, 10단계 체인은
800바이트 수준으로 현행 예산 내 수용 가능하다. 다만 경로가 여러 개 존재할 때(그래프
분기) 모두 나열하면 바이트 폭발 위험이 있으므로 "최단 경로 1개 + 대안 경로 수 표시"
전략이 필요하다.

**구현 복잡도**: 현재 스냅샷(`Vec<ExtractedFile>`)에는 호출 관계가 인덱싱되어 있지
않다. 체인 탐색 전에 `discover_callees` 방식으로 전체 호출 그래프를 온-디맨드 구성해야
하고, 이는 파일 수에 비례한 disk read를 유발한다. 대형 저장소에서 첫 호출 지연이
수 초에 달할 수 있다. 인덱스에 호출 그래프를 사전 빌드해 저장하는 방식은 인덱서
변경, 직렬화 형식 변경, 재인덱스를 수반한다 — 범위가 크다.

**결론**: 에이전트 경험을 가장 크게 개선하는 방향이지만, 구현 범위가 가장 넓고
온-디맨드 그래프 구성의 지연 문제가 해결되어야 실용적이다.

### (c) 현행 유지 — 복잡 과제를 예외로 인정

**현행 비용**: t4류 과제는 25~33턴. 다른 과제 중앙값 7~9턴 대비 약 3배 초과. 그러나
t4는 "RPC 분기 → 실행" 같은 다단계 아키텍처 이해 과제로, 코드베이스 전체를 이해해야
하는 특수 케이스에 속한다.

**다른 항목과의 투자수익 비교**:
- 01(리터럴 줄 번호): 구현 범위 좁음, t5·t6류 전체에 영향, 에이전트 확인 read 제거
- 03(타 언어 벤치마크): 구현 없음, 현행 한계 수치화만
- 04(본 항목): 구현 범위 중~대, t4류 특수 케이스에만 영향

12개 과제 중 t4 1개(또는 t4류)를 위한 전용 개발 비용과, 이미 9.5턴 중앙값을 달성한
상황에서의 우선순위를 고려하면, 현행 유지가 단기적으로는 합리적이다.

**결론**: 기술 부채는 남지만, 다른 항목 대비 레버리지가 낮다. 벤치마크에 t4류를
추가할 때마다 중앙값이 왜곡되므로, 이를 별도 "복잡 흐름 과제" 지표로 분리해
측정하는 것이 현실적 선택지다.

## 권장안

**방향 (a) — 디스패치형 함수에 한해 depth-2 callee 확장을 권장한다.**
단, 레이블링 방침을 먼저 확정하고 소규모 파일럿(surrealdb t4 과제 고정 후 측정)으로
노이즈 실효성을 검증하는 조건부 권장이다.

**이유**:
- 추가 워크스페이스 walk가 없어 응답 시간 영향이 최소화 — 스캔 비용 0, disk read
  증가 N회(N ≤ `callee_list_cap` = 5)에 그친다.
- 에이전트가 depth-1 callee 각각에 대해 별도 search 호출을 하는 현행 패턴을 1회
  응답으로 흡수할 수 있다 — 체인 1단계당 약 2~3턴 절약 예상.
- 방향 (b)보다 구현 범위가 명확히 작으며, 현행 `discover_callees` 함수를 조건부
  재호출하는 형태로 기존 아키텍처를 유지한다.

**핵심 트레이드오프 3줄**:
1. **비용**: 추가 스캔 없음, depth-2 대상 심볼당 file read 1회 추가 — 수용 가능.
2. **노이즈**: 이름 매치 오차가 depth마다 누적 — depth-2 항목 전체에 명시적 `(approximate, via {callee})` 레이블 필수이며, 이를 생략하면 에이전트 오판이 증가할 수 있다.
3. **효과 불확실성**: "디스패치형" 휴리스틱이 t4류 과제의 병목 지점(RPC handler 등)을 실제로 포착하는지는 파일럿 측정 전까지 미검증이다.

## Open Questions

1. **디스패치형 판별 조건 확정**: "depth-1 callee 수 ≥ 3" 조건과 "`match` arm 수
   ≥ 4" 조건 중 어느 것을, 또는 두 조건의 AND/OR 조합을 사용할지 — 트레이드오프는
   recall(더 많은 함수를 depth-2 확장) vs. precision(실제 디스패치 함수만 확장).

2. **depth-2 레이블 형식**: `(approximate, via {callee-name})` 인라인 레이블과
   별도 섹션(`_calls (depth 2, via {callee-name}):_`) 중 어느 형식이 에이전트 인용에
   더 적합한지.

3. **파일럿 측정 선행 여부**: 구현 전 surrealdb t4 과제를 고정 벤치마크에 추가해
   현행 25~33턴의 정확한 원인 분포(callee 추적 vs. 확인성 read vs. 기타)를 수치화할지,
   아니면 구현 후 측정으로 효과를 한번에 확인할지.

4. **annotation_sub_budget 조정 필요 여부**: depth-2 항목 추가로 주석 블록이 커지면
   현재 8192바이트 상한이 조기에 소진될 수 있다 — 기본값 유지 vs. 증가(예: 12288)
   여부를 결정해야 한다.
