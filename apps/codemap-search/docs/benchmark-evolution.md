response-diet)을 거치며 **현재 설계가 왜 이런 모습이 됐는가**를 설명하는 문서다.

영역) → 검증(재측정 핵심 수치)" 패턴으로 서술하며, 기존 회고록 3종을 통합 대체한다.

## 캠페인 1 — SurrealDB (v3→v7): search가 답을 싣게 만들기

codex CLI(gpt-5.5) 단일 arm, surrealdb(Rust, 약 2,700 파일), 6과제 × 2rep,
측정→수정 5회차. 과제 정의·ground truth 표(t1~t6)는
`bench-2026-06/tasks-surrealdb.md`가 단일 출처다. 에이전트의 검색 행동을
**발견 → 신뢰 → 인용** 3단계로 보고 각 단계의 병목을 차례로 제거했다.

### 1.0 전제 — MCP ToolAnnotations (`readOnlyHint`)

`readOnlyHint: true`가 없으면 codex는 비대화형 실행에서 모든 MCP 호출을 자동
취소한다(`requires_mcp_tool_approval`). 이 한 줄 수정 전에는 어떤 개선도 측정
자체가 불가능했다.

### 1.1 발견 — matched-symbol fallback 완화 (index.rs)

- **관찰**: detail 응답 24개 중 23개가 스니펫·caller 주석 없는 fallback이었다.
  matched-symbol 선택이 "모든 쿼리 단어가 한 심볼에 매치"를 요구해, GPT의 다단어
  쿼리("Transaction put_tb definition")는 글루 단어("definition") 하나 때문에 거의
  항상 fallback으로 떨어졌고, fallback은 라인 순서 상위 20개 심볼명만 나열했다 —
  L3049의 `put_tb`는 목록에 보이지도 않았다.
- **가설**: 전단어 일치 요구가 과도하다 — 식별력 있는 단어의 정확 일치만으로
  매치를 인정해야 한다.
- **수정**: (a) 8자+ 또는 다토큰 심볼명과 정확 일치 시 무조건 매치, (b) 3단어
  이상 쿼리에서 절반 이상 매치 시 승격, 정확 일치 우선 정렬
  (`search_detail_symbol_limit` 컷 보호).
- **검증**: v5에서 주석 활성화 1/24(4%) → 89.5%.

### 1.2 v5 중앙값 역설과 rocksdb/mod.rs 32.6KB 독식 (v5→v6)

- **관찰 1 — 풍부해진 응답이 턴을 늘렸다**: v5에서 정답 노출은 개선됐는데 중앙값
  턴이 9 → 12로 **올랐다**. 응답이 풍부해지자 에이전트가 더 철저히 검증했기
  때문이다(t6: v3는 발생 지점 1곳 예시, v7은 KV 백엔드 5종 전수 열거). 턴 수만
  보면 품질 향상이 회귀로 보인다 — "정답이 처음 노출된 턴"과 "답의 완전성"을
  같이 보는 측정 교훈이 여기서 나왔다.
- **관찰 2 — 한 파일의 바이트 독식**: t6 쿼리 응답이 33KB로 비대했다. 부분 매치
  승격이 독스트링만으로 발화해, 독스트링이 쿼리 단어를 스치는 fn 11개가 한 파일
  (rocksdb/mod.rs)에서 32.6KB를 렌더링해 바이트 예산을 독식했고, 더 관련성 높은
  파일 2개가 잘려나갔다.
- **가설**: 부분 승격에는 심볼 이름 쪽 증거가 최소 1개는 있어야 한다.
- **수정**: 심볼명/분할 토큰에 최소 1개 쿼리 단어가 매치해야 부분 승격이 발화하는
  이름 증거 게이트(index.rs).
- **검증**: t6 응답 33.2KB → 15.6KB, 5/5 파일 렌더 복원.

### 1.3 인용 병목 — 줄 번호 스니펫 + 라벨 명확화 (v6→v7, 단일 변경 최대 낙폭)

- **관찰**: v6 분석에서 정답 노출은 12/12 모두 턴 1인데 중앙값은 12.5였다. 턴이
  새는 곳은 발견이 아니라 **인용 단계** — 과제가 정확한 줄 번호를 요구하는데
  스니펫에 줄 번호가 없어 에이전트가 이미 본 범위를 read로 다시 읽었고, caller
  라벨 "(approximate, name-match only)"가 정확한 file:line까지 재검증하게 만들었다.
- **수정**: 스니펫을 read와 동일한 `  3049→ …` 포맷으로 렌더링. 라벨을 "(file:line
  positions exact; name-match attribution approximate)"로 교체 — 근사인 것은 동명
  정의 간 귀속뿐이고 위치는 정확하다는 사실을 라벨이 그대로 말하게 했다.
  인스트럭션에 "줄 번호와 caller 위치는 재읽기 없이 바로 인용하라"를 명시.
- **검증**: 중앙값 12.5 → 9.5. 에이전트는 라벨을 문자 그대로 읽는다 — 문구의
  정밀함이 곧 성능이다.

### 1.4 발견 — enum variant 추출 + 문자열 리터럴 인덱싱 (parser.rs, index.rs)

- **관찰**: variant(`TransactionReadonly`)가 심볼이 아니어서 에러 정의 파일
  (err.rs)이 search로 발견 불가 → grep 우회 4~5턴. 설정 기본값("8000")·에러
  메시지 같은 문자열 리터럴도 인덱스에 없었다.
- **수정**: Rust `enum_variant`를 kind `variant`로 추출하고 owner(소속 enum)를
  부여. tantivy `literal` 필드 신설(부스트 1.0 — 심볼 4.0·독스트링 2.0보다 낮게,
  256자 캡). 포맷 버전 범프로 재인덱스 강제.
- **검증**: err.rs가 search 1위, t5류(기본 포트)가 첫 search 응답에서 발견.

### 1.5 발견 — caller 스캔 이름별 예산 (callers.rs)

- **관찰**: `scan_cap`(500)이 전 스캔 이름 공유의 전역 예산이라, walk 초반의 핫한
  이름이 소진하면 뒤쪽 파일의 호출처가 0건 — 실제 호출처 19곳이 있는 `put_tb`가
  "no direct caller observed"로 출력됐고 절단 표시도 없었다.
- **수정**: 예산을 이름별 분배(`scan_cap / names`, 하한 25)하고 절단을 이름별로
  추적·표시.
- **검증**: t2(정의+호출처)가 search 1콜에 통째로 실린다.

### 1.6 안정성 — 동시 기동 인덱스 wipe 경쟁 (index.rs)

- **관찰**: 신규 repo에서 서버 2개가 동시 기동하면 둘 다 포맷 sidecar 부재 →
  wipe+재생성 경로를 타며 **서로의 인덱스 파일을 삭제**했다
  (`test_mcp_parallel_requests` 플레이크의 원인).
- **수정**: 빈 인덱스 디렉터리(meta.json 없음)는 wipe 없이 제자리 생성, 파괴적
  복구 전 1회 재시도.
- **검증**: `test_mcp_parallel_requests` 플레이크의 원인 제거. 이후 캠페인의
  동일 repo 병렬 실행(동시성 8) 무사고는 이 수정과는 별개 근거 — 사전 인덱싱 +
  비치명 LockFailure 강등(사전 인덱싱된 repo에서는 wipe 경로 자체가 타지
  않는다) — 에 기반하며, 실측 근거는 playbook §4 참조.

### 1.7 무효 회차 v4 — 프롬프트 재구성 사고

v4는 프롬프트 재구성 실수로 도구 유도 문구가 빠져 search가 182콜 중 3콜만 쓰인
오염 회차다(측정 무효). "프롬프트는 tasks JSON에서 jq로 추출, 한 글자도 수정
금지" 규칙(playbook §2)이 여기서 나왔다.

캠페인 1 종료 시점: 정답 파일 첫 search 응답 포함 9/12 → 12/12, 잔여 한계는
t4류 다단계 흐름 추적(25~33턴, `design-callchain-tracing.md`로 분기).

## 캠페인 2 — ollama + ClickHouse (C/C++): 2-arm 160 에피소드

claude sonnet + codex gpt-5.5의 2 arm × 10과제 × 2rep × 2repo = 80 에피소드/회차.
멀티 모델 오케스트레이션(fable 계획·검증 / sonnet 측정 / opus 수정) 첫 도입.

### 2.1 측정 전 게이트 — 참조 반환 심볼 3,053줄 누락 (parser.rs, 블로커)

- **발견 경위**: 벤치마크 시작 전 fable의 최종 검증이 실저장소 인덱싱 스모크로
  적발했다 — opus 적대 리뷰 **2회를 통과한** 코드였다. tree-sitter-cpp의
  `reference_declarator`는 `declarator` 필드가 없는(positional child) 노드라 필드
  기반 탐색이 항상 None → `int& getref()`, `T& operator=` 류가 통째로 미추출.
  ClickHouse src 기준 정의 라인 약 3,053개.
- **의미**: 수정 없이 측정했다면 ClickHouse 과제 다수가 "제품 결함인지 검색
  한계인지" 분리 불가인 채로 오염됐다 — "측정 전 실저장소 인덱싱 스모크" 게이트가
  여기서 굳어졌다. 같은 게이트에서 함수-로컬 vexing-parse 가짜 심볼(전체 심볼의
  8.5%)과 in-class private 인라인 정의의 exported 오판도 함께 수정 → 포맷 `v6`.

### 2.2 iter1 관찰 3건 → 수정 → iter2 검증 (77/79 → 80/80)

- **grep 기본 output_mode** — 관찰: grep 호출의 95.1%(116/122)가 기본값
  (files_with_matches)을 그대로 사용, 줄 번호 없는 파일 목록만 받고 search/read
  재탐색 루프에 빠졌다. 캠페인 1 교훈("줄 번호가 없으면 이미 본 범위를 다시
  읽는다")이 grep에서 재현된 것. 수정: 기본값을 content(+줄 번호)로 전환. 검증:
  c7(에러코드 throw 추적)이 50턴 컨텍스트 한계 partial → **3턴 correct**.
  에이전트는 압도적으로 최소 인자로 도구를 부른다 — 기본값이 가장 뜨거운 코드
  경로다.
- **read 관용 파라미터** — 관찰: claude 계열이 `path`/`file`/`start_line`/
  `end_line`을 관용적으로 보내 52건 하드 에러(codex는 0건 — 스키마 준수 성향이
  모델별로 다르다). 1차 수정은 별칭 수용. 검증: iter2에서 에러 52 → 5건. 잔여
  원인은 2.3 참조.
- **owner 토큰 인덱싱** — 관찰: `get`처럼 짧고 흔한 이름은 BM25 식별력이 없어
  "StorageFactory get" 한정 쿼리로도 미발견(c8: search 10회 실패, partial).
  수정: 멤버 심볼 문서에 owner 이름·분리 토큰을 추가 인덱싱(`v7`). 검증: c8이
  **search 2회 즉시 발견**, correct.

### 2.3 오진과 정정 — read 결함의 진짜 근본 원인

- **오진**: sonnet의 iter2 분석은 잔존 read 결함을 "별칭 미지원" 문제로 진단했다.
- **정정**: fable의 라이브 바이너리 검증에서 진짜 원인이 드러났다 — 에이전트가
  숫자를 JSON 문자열(`"228"`)로 보내면 `as_u64()`가 **조용히 None**을 반환해 줄
  범위가 통째로 탈락, 파일 전체가 렌더링됐다(102KB 한도 오류 연쇄). 오진대로
  별칭만 추가했다면 최종 수정 라운드는 아무것도 고치지 못했다.
- **수정**: 공용 `lenient_usize` 코어션 + 별칭 맵으로 일반화. **별칭 두더지잡기가
  아니라 코어션 레이어가 올바른 설계** — 이 캠페인의 핵심 도구 설계 교훈.
- 같은 정정 패스에서 first_answer_turn 하락(0.8→0.4)을 회귀로 해석한 것도
  뒤집혔다 — ToolSearch(하니스 메커니즘)를 도구 호출로 센 메트릭 아티팩트였고
  보정 시 0.9→0.9 동일. **점수는 전부 정직했고 서사만 틀렸다.**

### 2.4 모델 역할에서 얻은 것

sonnet은 현상 발견(grep 기본값·read 에러)에 정확했지만 코드 수준 root-cause에서
사실 오류를 냈고, opus는 정확한 진단이 주어졌을 때만 정확했으며, 블로커 적발과
오진 정정은 모두 fable 게이트에서 나왔다. 여기서 "물량은 sonnet, 명세된 수정은
opus, 진실 판정은 fable" 배치와 "정정 패스를 분석 직후에 둔다" 규칙이 playbook
§5·§6으로 굳어졌다. 효율 검증 핵심값: claude 중앙값 턴 −53%(9.5→4.5), 응답
바이트 −51~58%(상세는 `benchmark-workflow.md` §8 캠페인 2 절).

## 캠페인 3 — django + strapi (ds-iter1): hold-out 시험

- **관찰(문제의식)**: 캠페인 2의 수정은 전부 in-sample이었다 — c8에서 진단한
  수정을 c8로 검증했다.
- **가설**: 수정이 일반적이라면 미접촉 언어(Python/TypeScript)·미접촉 저장소·
  더 어려운 과제에서도 효과가 유지된다.
- **검증**: 제품 무변경(`v7-owner-tokens-indexed` 그대로)으로 80 에피소드 측정 —
  80/80 correct, first_answer_turn 평균 ~1, read 파라미터 에러 52건(캠페인 2
  iter1) → **0건**. 캠페인 1에서 25~33턴 걸리던 다단계 흐름 부류가 hard 평균
  6.1~9.0턴. **hold-out 성립.**
- **단서**: 일부러 어렵게 만든 과제 세트가 다시 포화(전 arm 100%) — 정확도 축은
  변별을 못 했고, 변별은 효율 축(도구 믹스·턴·바이트)에만 남았다.

운영 측면에서는 재작성된 하니스(에피소드 병렬, 동시성 8)의 첫 실측 — 80 에피소드
wall clock 약 5분(직렬 운영은 2.4~4.0h), 동일 repo 병렬 무사고. 절차 사건 3건
(조기 회차 잔재, 재채점 역할 오배치, 커밋 핀 재미이행)은 전부 본실행 데이터
무영향으로 playbook 규칙(§4-8, §6, §1)에 환류됐다.

본실행 이후는 사전 확정한 8단계 계획(당시 `post-run-plan.md`, 이 절로 흡수)대로
진행했다: ① 채점 입력 생성(`harness/build-scoring-batches.sh`, repo×arm×rep
8배치) → ② 채점(sonnet ×8 병렬 — rubric을 기계적으로 적용, StructuredOutput
스키마 강제, 기입은 오케스트레이터가 jq로 일괄 수행. 저장소 재탐색으로 정답
재구성·answer_text 의역·rubric에 없는 정상참작 금지) → ③ 재채점 게이트(fable,
표본 40) → ④ 집계(arm×repo×난이도) → ⑤ 회차 분석(실패를 제품/하니스·채점/모델
한계로 분류, transcript 인용 의무) → ⑥ root-cause 정정 패스(fix_hypothesis를
라이브 바이너리로 재현 검증 — 캠페인 2 오진 재발 방지) → ⑦ 루프 결정(정답률
100% 미만 **그리고** 입증된 제품 결함일 때만 수정 라운드) → ⑧ 종료 정리(문서·
자산 동기화). ds-iter1은 ⑦의 재루프 조건 미충족(80/80)으로 수정 라운드 없이
종료했다. 진행 중 결정 1건 — 제품 무변경 상태로 조기 시작돼 19/80에서 중단된
ds-iter2 잔재는 run-matrix의 멱등 재개(metrics 있으면 skip) 특성상 향후 수정 후
본실행과 바이너리가 섞일 수 있어 회차째 삭제했다(수정 라운드가 생기면 같은
회차명을 새로 시작).

## 캠페인 4 — baseline arm (ds-base1): 부가가치의 직접 측정

- **관찰(문제의식)**: 세 캠페인 연속 "빌트인 도구 대비 얼마나 나은가"가 미답.
- **검증**: 동일 과제·하니스에서 빌트인 전용 arm 2종 측정 — baseline도 **80/80
  correct**(함정 포함). 이 난이도의 자작 내비게이션 과제로는 MCP 부가가치를
  정확도로 증명할 수 없음이 확정됐다.
- **효율 신호는 CLI별 반대 방향**: claude에서는 탐색 구조 개선(턴 −11%,
  first_answer 1.56→1.15)이되 응답이 무겁고(도구 결과 바이트 +73%), codex에서는
  턴 동등이되 hard 과제 셸 출력 폭주가 억제됐다(평균 바이트 −45%, 꼬리 효과).
  "codemap-search가 토큰을 줄인다"는 일반 주장은 이 데이터로 성립하지 않는다.
- **다음 루프의 가설 도출**: claude MCP의 비용 역전 원인은 응답 크기다 → 응답
  다이어트가 제품 백로그 1순위로 승격. baseline 결과는 제품 수정의 영향을 받지
  않으므로 이후 루프의 고정 기준선이 됐다.

## 캠페인 5 — 응답 다이어트 (ds-iter2 → ds-iter3): 에이전트는 보상한다

### 5.1 루프 1 (ds-iter2) — P1~P6 렌더 수정

ds-base1 분석에서 도출한 렌더 계층 수정 6건(인덱스 포맷·스캔 예산·랭킹·코어션
무접촉): ① 쿼리 매칭 심볼 앵커 렌더링(Tier-1 통식별자 정확 일치 우선, 비매칭
컨테이너는 2~3줄 요약), ② 동일 이름 caller 블록 dedup("same as above"), ③ 정의
≥5 이름 caller 생략(`caller_omit_def_threshold`), ④ 꼬리 목록 50→12, ⑤ grep 기본
제외에 `.yarn/`·minified 번들 추가, ⑥ 도구 설명에 재확인 불필요 예시.

적용 전 라이브 A/B(보존한 수정 전 바이너리와 동일 쿼리 비교)에서 결함 2건을
사전 적발·정정했다: (a) 토큰 교집합 매칭 과완화 — `select` 부분 토큰이
`get_select` 등에 풀 스니펫 부여 → Tier-1 통식별자 일치 우선으로 정밀화,
(b) caller dedup이 스캔 순서 기준이라 원본 블록 없는 dangling "same as" 발생 →
렌더 시점·렌더 순서 기준 + 실출력 후에만 참조 허용으로 이동.

### 5.2 ds-iter2 검증 — 응답은 줄었는데 claude는 보상했다

- **관찰**: 동일 쿼리 A/B 절감은 실현(−22~53%)됐고 codex는 에피소드 총량까지
  순개선(−18%)했는데, **claude 에피소드 총량은 오히려 +16%**(23.8→27.6KB).
- **도구별 분해가 원인을 드러냈다**: search 합은 745K→780K로 거의 그대로인데
  **read +53%(120K→184K), grep +74%(53K→92K)** — 앵커링이 한 줄 스텁으로 강등한
  비매칭 심볼 맥락을 claude가 통파일 read로 다시 가져왔다(s7 최대 응답이 9.0KB
  search → 30.4KB 통파일 read로 교체).
- **교훈 8**(`benchmark-workflow.md` §8의 설계 교훈 7가지에 잇는 8번째): 캠페인 1의
  "풍부한 응답이 턴을 늘린다"(1.2)의 역방향 짝 —
  **빈약해진 응답은 보상 read를 늘린다.** 응답 크기 최적화는 응답 단위가 아니라
  에이전트의 보상 행동까지 포함한 에피소드 총량으로 판정해야 한다.

### 5.3 루프 2 (ds-iter3) — 보상의 3개 기전을 흡수

fable의 transcript 재진단이 iter2 회귀를 3개 기전으로 분해했고, 셋을 묶어 수정:

1. **시그니처 축약**(s8형 — 스텁 맥락 손실→search 반복): 강등 심볼을 한 줄 스텁
   대신 Tier-2는 시그니처+최대 3줄, 비매칭은 시그니처 1줄 + 생략 마커로 렌더.
2. **read 별칭 키 정규화**(s7형 — 제품 결함): `startLine`(카멜)이 별칭 미등록으로
   조용히 무시돼 30.4KB 통파일 렌더 ×2. 별칭 매칭을 키 정규화(소문자화+`_`/`-`
   제거)로 일반화 — 2.3 코어션 교훈의 완성.
3. **파일당 앵커 풀 스니펫 상한**(d9형 — `save`·`send` 같은 흔한 이름 다발 앵커):
   `search_anchor_snippet_limit`(기본 3) 신설, 초과 앵커는 축약으로 강등.

구현 중 A/B로 추가 적발: 심볼 초과 **폴백 분기**에 앵커 캡·축약이 미적용이라
d9가 개선 0이던 것을 양 분기 공유 렌더 함수로 통일(d9 프로브 30.1KB→14.1KB).

**검증**: claude 에피소드 총량 23.8→27.6→**19.5KB**(read 보상 흡수: 193→135K),
codex −14% 유지, 정답률 80/80. iter2 폭증 과제 전부 회복(s7 92.4K→39.1K 등).
유일한 역방향은 claude first_answer 평균 1.15→1.70(전건 도달) — 차기 추적 항목.

### 5.4 사후 적발 — 도구 설명의 d7 정답 단편 누출 (오염 포렌식)

- **관찰**: 캠페인 종결 후 "패치가 벤치마크 특정적인가" 점검에서, 루프 1의 P6이
  넣은 도구 설명 예시 `compiler.py:1595→ def execute_sql(...)`가 **d7 rubric
  4단계 정답과 정확히 일치**함을 적발. 출처는 오케스트레이터가 d7 진단 사례를
  발주 브리프 예시로 그대로 옮긴 실수 — iter2·iter3 측정 내내 모든 search 도구
  설명에 노출됐다.
- **포렌식**: 3개 회차 × d7 4에피소드 = 12건 전수에서 `1595`가 답변 이전에 도구
  결과에 2~5회 선행 등장 — 맹목 인용(힌트 베끼기) 0건, 전부 자력 발견.
- **정정·재검증**: 예시를 중립 문구로 교체(렌더 바이트 무변화, 전 과제 정답과의
  충돌 전수 검사 통과) 후 무힌트 재실행(ds-d7check): 4/4 correct, 함정 0 —
  **정확도 결론은 오염 무관**. 그러나 claude d7 턴이 11~12로 복귀(힌트가 있던
  회차는 ~7) — **d7 턴 개선은 렌더 효과가 아니라 힌트 효과였다.** 보정 결과
  iter3 claude 평균 턴 4.05→4.25("턴 소폭 개선" 주장 철회), 평균 바이트는
  19.5→18.4KB로 절감폭이 오히려 확대(−22.5%).
- **교훈**: **제품이 contestant에게 출하하는 모든 문자열(도구 설명·에러 메시지·
  예시)은 측정 표면이다.** tasks JSON의 expected file/line/symbol과의 기계 diff가
  워크플로우 불변 조건(`benchmark-workflow.md` §4 원칙 6)으로 추가됐다.

## 채점 신뢰성 — gate-verdict 시리즈가 말해주는 것

매 회차 어려운 과제 중심 표본 40건을 "sonnet이 인용 줄 전수 원본 대조·rubric
체크리스트를 기계 수집 → fable이 판정"하는 분업으로 재채점했다
(원본: `bench-2026-06-django-strapi/gate-verdict*.md` 4종).

- **ds-iter1 / ds-base1 / ds-iter2: overturn 0.** 인용 대조 ~300건 수준에서 나온
  불일치 플래그(±1 보조 인용, 스니펫 줄 라벨 전사 드리프트, 의역)는 전부 rubric
  비요소로 채점 비영향. 도구가 반환한 줄 번호는 전 건 정확 — ±1 전사는 모델 표기
  습관이다(줄 라벨 ±0 rubric을 만들면 변별 요소가 될 수 있음). 수집기의 요건
  과잉 해석은 fable 판정에서 기각됐다.
- **ds-iter3: 시리즈 최초 overturn 2건 (partial→correct)** — 둘 다 모델 답안
  회귀가 아니라 **1차 채점자 오류**였다. s9-r2는 답안에 명백히 존재하는 섹션
  (L3553 인용 + 스니펫)을 누락으로 판정한 사실 오인, s10-r1은 rubric의 "추가 언급
  감점 없음" 조항 미적용. 두 건 모두 "채점자가 더 엄격" 방향이고, 같은 답안
  스타일이 이전 회차들에서는 correct였다 — 즉 채점 변동성이다.
- **교훈**: 기계 판정 가능한 rubric + 수집·판정 분리는 두 캠페인 연속 불일치 0을
  만들 만큼 견고하지만, 채점자 엄격성 편차까지 막지는 못한다. → **1차 채점
  rationale에 rubric 조항 인용을 의무화**(차기 채점 프롬프트 보강 항목).
