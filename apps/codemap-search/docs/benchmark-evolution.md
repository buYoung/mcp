# codemap-search 벤치마크 진화 — 왜 만들었고 어떻게 지금 모습이 됐는가 (2026-06)

이 문서는 codemap-search MCP를 왜 측정하기 시작했는지(프롤로그 — 사전 벤치마크의 반증)부터, 2026-06 에이전트 캠페인 5종(SurrealDB, C/C++, django+strapi hold-out, baseline, response-diet)을 거쳐 **현재 설계가 왜 이런 모습이 됐는가**까지를 단일 서사로 설명한다. 측정 수치 테이블만으로는 알 수 없는 인과 — 어떤 관찰에서 어떤 가설을 세웠고 무엇을 고쳤으며 효과가 어땠는지 — 를 "관찰(측정에서 본 문제) → 가설 → 수정(코드 영역) → 검증(재측정 핵심 수치)" 패턴으로 서술하며, 기존 회고록 3종과 사전 벤치 결과 문서(축 A/B·overview 고도·crossover)·회차 분석·재채점 게이트 판정 기록·개선 백로그 문서의 인과·판정·우선순위를 통합 대체한다.

읽는 법 — 역할 분담 (2문서 + 설정 레퍼런스):

- **이 문서 (`benchmark-evolution.md`)**: 인과 서사의 단일 출처 — 기원(프롤로그), 캠페인 1~5, 채점 신뢰성(재채점 게이트 판정 기록 포함), 제품 개선 백로그. 수치는 중복 게재하지 않고 검증 핵심값 1~2개만 인용한다.
- **`benchmark-workflow.md`**: 측정의 단일 기준 문서 — 측정 정의·단계 구성·모델 배치(§1~§3), 실행 원칙·실측 함정 표·운영(§4), 자산 포인터(§5), 메트릭 산출 정의(§6), 고정 실행 명령·격리 플래그(§7), 사전 벤치마크 축 A/B 수치(§8), 캠페인별 결과 표·한계 목록(§9), 재현 가이드(§10).
- **`configuration.md`**: 제품 설정 키 레퍼런스 — 캠페인이 도입한 `caller_omit_def_threshold`·`search_anchor_snippet_limit` 등의 단일 출처.

## 프롤로그 — 왜 이 벤치마크 연작을 시작했나 (2026-06-09~10)

출발점은 회사 대형 코드베이스에서의 체감이었다. codemap-search는 인터랙티브 세션에서 grep+read 대비 "압도적"이라는 주관적 경험을 줬고, 그 경험이 객관적으로 재현되는지 확인하기 위해 OSS 코퍼스에서 사전 벤치마크 2축을 돌렸다. 결과는 정반대였다 — 그리고 그 반증이 이 연작 전체의 출발점이다.

### 0.1 축 A — 병목은 성능이 아니라 컨텍스트다

- **측정**: 4티어(fd ~7k → vue-core ~128k 지원 LOC)에서 인덱싱·메모리·도구 지연·응답 크기를 ground truth 없이 특성화했다. 성능은 어디서도 병목이 아니었다 — 128k LOC 색인 0.63s, 전 도구 warm p95 ≤ 14ms.
- **관찰**: 진짜 비용은 지연이 아니라 응답 크기였다. 128k 코퍼스의 root overview가 761KB 텍스트 — 지연(수 ms)은 싸지만 에이전트 컨텍스트에 통째로 들어가는 순간 토큰 비용이 모든 것을 지배한다.
- **함의 두 가지**: ① 이후 모든 에이전트 측정에 반환 바이트를 채점 축으로 포함한다 — `mcp_response_bytes_total` 표준 메트릭과 contestant 토큰의 1급 메트릭 승격(workflow §6)이 여기서 출발했고, 같은 문제의식이 캠페인 5 응답 다이어트까지 닿는다. ② overview의 역할 자체를 재설계해야 한다(→ 0.2). 수치 표는 workflow §8-2가 단일 출처다.

### 0.2 overview 고도 보정 — 과신은 옮겨 다닌다 (첫 관찰→가설→수정→재측정)

캠페인에 앞서 "관찰→가설→수정→재측정" 패턴이 처음 작동한 곳은 overview였다:

- **관찰**: 행동 관찰(fd, codemap 도구만 허용)에서 에이전트는 root overview 덤프 1개(20,688B — 597줄의 97%가 per-symbol 라인범위)를 전적으로 신뢰하고 search를 0회 사용했다. overview가 완전한 심볼 인덱스로 동작하며 search를 대체하고 있었다.
- **가설·수정 1 — root 슬림화**: root에서 per-symbol 라인 위치를 제거하고 디렉터리 + 파일당 1줄 요약으로 바꾸면(20,688B → 1,265B, −94%) 위치 탐색이 search로 넘어갈 것이다.
- **재관찰 — 과신의 folder 이동**: 틀렸다. R1 재관찰(이하 R1/R2 = 수정 1/2 후 행동 재관찰 라운드)에서 에이전트는 root+folder 덤프 2개를 신뢰했을 뿐 search는 여전히 0회 — folder가 라인범위를 주는 한 folder가 locator다.
- **수정 2 (사용자 결정) — folder 라인범위 제거**: folder에서도 라인범위를 제거해 정확 위치를 search/grep으로 강제했다. R2 재관찰에서 비로소 grep 5회·search 1회 — overview=orient, search/grep=locate 분리가 처음 실현됐고, essential anchor(데이터셋이 필수로 지정한 정답 위치)는 전부 정확히 특정돼 탐색 품질 손상도 없었다.
- **크로스오버 스윕으로 결정 잠금**: "어느 인덱스 크기부터 라인범위가 손해인가"를 티어별 강제형 A/B(n=3)로 측정 — 라인범위가 토큰상 이득인 구간은 index ≲ 0.13MB(약 7k LOC)뿐이고 그 위로는 크기에 비례해 악화(−7.5% @1.4MB)했다. index 크기 임계 하이브리드는 검토 후 폐기했다 — tantivy 인덱스는 첫 search 때 lazy 빌드되므로 overview 시점의 인덱스 크기는 신뢰할 수 없고, overview 동작이 search 이력에 의존하면 결정성이 깨진다. 라인범위 항상 숨김(R2)으로 고정.

여기서 굳은 두 가지가 연작 전체를 관통한다: **도구의 출력 형태가 곧 에이전트의 행동을 결정한다**는 것, 그리고 **수정은 반드시 재측정으로 닫아야 한다**는 것 — 수정 1만 보고 멈췄다면 "root 슬림화로 search 사용을 유도했다"는 틀린 결론이 남았을 것이다.

### 0.3 축 B — baseline grep+read가 Pareto 최적이라는 반증

arm 3종은 baseline(빌트인 도구만)·pure(codemap 도구만)·additive(양쪽 모두) — 구성 정의는 workflow §8-3.

- **1라운드 — edit-site localization** (fd·ripgrep, 3 arm × 3 rep): codemap은 빌트인에 추가 정확도를 주지 못했고(보정 후 recall 동률, 가장 비자명한 앵커도 baseline 3/3·pure 3/3 동률), 그 동률을 토큰 약 1.5×로 샀다. 빌트인+codemap 동시(additive)는 가장 어려운 지점에서 오히려 탐색이 흩어져(앵커 적중 1/3) Pareto 열위 — "열등"이 아니라 "산만".
- **2라운드 — context-gathering** (scrapy, 174파일): "동적 로딩 indirection은 grep이 못 따라가 codemap이 이긴다"는 가설이 반증됐다. 에이전트는 순수 grep이 아니라 grep→read→import·호출 추적이라, 이 규모에선 "읽어서 조립"이 충분히 싸서 baseline이 recall 1.00으로 호출 체인을 완전 포착했다. codemap은 또 +56% 토큰에 동률.
- **transcript 포렌식이 비용의 실재를 확정**: 보고된 토큰은 context-inclusive 측정이었고, cost-weighted로는 baseline 대비 1.9× — raw 지표는 비용을 과소평가했지 과대평가가 아니었다. "멀티 corpus 스키마 고정비가 비용을 부풀린다"는 측정 아티팩트 가설도 반증됐다.
- **3개 task class 전부에서 baseline이 Pareto 최적 — 회사 경험("압도적")은 재현되지 않았다.** 수치·스코프 경계·원자료는 workflow §8-3·§8-4가 단일 출처다.

### 0.4 반증이 남긴 가설 — 에이전트 캠페인으로의 전환

축 B의 정확한 결론은 "codemap 무용"이 아니라 "**이 규모(≤174파일/≤128k LOC)·이 과제군(keyword-greppable, 단발 sub-agent)에선 중복**"이다. 미검증 변수가 셋 남았다:

1. **규모(가장 유력)**: 이 규모에선 읽어서 조립이 싸다. grep hit·read 비용이 폭증하는 수천 파일·수십만 LOC 이상에서만 교차점이 날 수 있다.
2. **과제 설계**: edit-site가 검색 가능한 심볼·플래그명으로 특정되는 과제는 codemap의 가설적 강점 — 키워드를 모를 때의 구조적 내비게이션, BM25 개념·리터럴 검색 — 을 시험하지 않았다.
3. **측정 방식**: 단발 sub-agent의 context 비용 측정은 회사의 인터랙티브 멀티턴 경험과 다른 것을 잰다.

연작은 여기서 노선을 바꿨다. "codemap이 빌트인을 이기는가"라는 가로 비교를 잠시 내려놓고, **수천 파일 대형 저장소(SurrealDB ~2,700파일, ollama+ClickHouse, django+strapi)에서 멀티턴 CLI 에피소드로 에이전트의 실제 도구 사용을 관찰하고, 관찰→가설→수정→재측정 루프로 제품을 고치는 세로 개선 캠페인**으로 전환했다. 측정 방식도 갈아탔다 — 단발 sub-agent `<usage>` 캡처 대신 신선한 contestant CLI 세션 + 구조화 이벤트 스트림 jq 추출 + 기계 판정 rubric(사전 벤치가 캠페인 설계에 남긴 것의 전모는 workflow §8-5). 가로 비교 질문은 캠페인 4(baseline arm)에서 같은 자리로 되돌아온다 — 그리고 사전 벤치의 결론이 다른 저장소·다른 과제·다른 측정 방식으로 재확인된다.

## 캠페인 1 — SurrealDB (v3→v7): search가 답을 싣게 만들기

codex CLI(gpt-5.5) 단일 arm, surrealdb(Rust, 약 2,700 파일), 6과제 × 2rep, 측정→수정 5회차. 과제 정의·ground truth 표(t1~t6)는 코퍼스 결속 정답 키라 본 서술 문서에도, 저장소에도 남기지 않는다(데이터셋 제거됨 — `benchmark-workflow.md` §5) — 캠페인 1의 서사는 본 절과 `benchmark-workflow.md` §9-4가 단일 기록이다. 에이전트의 검색 행동을 **발견 → 신뢰 → 인용** 3단계로 보고 각 단계의 병목을 차례로 제거했다.

### 1.0 전제 — MCP ToolAnnotations (`readOnlyHint`)

`readOnlyHint: true`가 없으면 codex는 비대화형 실행에서 모든 MCP 호출을 자동 취소한다(`requires_mcp_tool_approval`). 이 한 줄 수정 전에는 어떤 개선도 측정 자체가 불가능했다.

### 1.1 발견 — matched-symbol fallback 완화 (index.rs)

- **관찰**: detail 응답 24개 중 23개가 스니펫·caller 주석 없는 fallback이었다. matched-symbol 선택이 "모든 쿼리 단어가 한 심볼에 매치"를 요구해, GPT의 다단어 쿼리("Transaction put_tb definition")는 글루 단어("definition") 하나 때문에 거의 항상 fallback으로 떨어졌고, fallback은 라인 순서 상위 20개 심볼명만 나열했다 — 정답 심볼 `put_tb`는 목록에 보이지도 않았다.
- **가설**: 전단어 일치 요구가 과도하다 — 식별력 있는 단어의 정확 일치만으로 매치를 인정해야 한다.
- **수정**: (a) 8자+ 또는 다토큰 심볼명과 정확 일치 시 무조건 매치, (b) 3단어 이상 쿼리에서 절반 이상 매치 시 승격, 정확 일치 우선 정렬 (`search_detail_symbol_limit` 컷 보호).
- **검증**: v5에서 주석 활성화 1/24(4%) → 89.5%.

### 1.2 v5 중앙값 역설과 rocksdb/mod.rs 32.6KB 독식 (v5→v6)

- **관찰 1 — 풍부해진 응답이 턴을 늘렸다**: v5에서 정답 노출은 개선됐는데 중앙값 턴이 9 → 12로 **올랐다**. 응답이 풍부해지자 에이전트가 더 철저히 검증했기 때문이다(t6: v3는 발생 지점 1곳 예시, v7은 KV 백엔드 5종 전수 열거). 턴 수만 보면 품질 향상이 회귀로 보인다 — "정답이 처음 노출된 턴"과 "답의 완전성"을 같이 보는 측정 교훈이 여기서 나왔다.
- **관찰 2 — 한 파일의 바이트 독식**: t6 쿼리 응답이 33KB로 비대했다. 부분 매치 승격이 독스트링만으로 발화해, 독스트링이 쿼리 단어를 스치는 fn 11개가 한 파일 (rocksdb/mod.rs)에서 32.6KB를 렌더링해 바이트 예산을 독식했고, 더 관련성 높은 파일 2개가 잘려나갔다.
- **가설**: 부분 승격에는 심볼 이름 쪽 증거가 최소 1개는 있어야 한다.
- **수정**: 심볼명/분할 토큰에 최소 1개 쿼리 단어가 매치해야 부분 승격이 발화하는 이름 증거 게이트(index.rs).
- **검증**: t6 응답 33.2KB → 15.6KB, 5/5 파일 렌더 복원.

### 1.3 인용 병목 — 줄 번호 스니펫 + 라벨 명확화 (v6→v7, 단일 변경 최대 낙폭)

- **관찰**: v6 분석에서 정답 노출은 12/12 모두 턴 1인데 중앙값은 12.5였다. 턴이 새는 곳은 발견이 아니라 **인용 단계** — 과제가 정확한 줄 번호를 요구하는데 스니펫에 줄 번호가 없어 에이전트가 이미 본 범위를 read로 다시 읽었고, caller 라벨 "(approximate, name-match only)"가 정확한 file:line까지 재검증하게 만들었다.
- **수정**: 스니펫을 read와 동일한 `  1234→ …` 포맷으로 렌더링. 라벨을 "(file:line positions exact; name-match attribution approximate)"로 교체 — 근사인 것은 동명 정의 간 귀속뿐이고 위치는 정확하다는 사실을 라벨이 그대로 말하게 했다. 인스트럭션에 "줄 번호와 caller 위치는 재읽기 없이 바로 인용하라"를 명시.
- **검증**: 중앙값 12.5 → 9.5. 에이전트는 라벨을 문자 그대로 읽는다 — 문구의 정밀함이 곧 성능이다.

### 1.4 발견 — enum variant 추출 + 문자열 리터럴 인덱싱 (parser.rs, index.rs)

- **관찰**: 에러 enum variant가 심볼로 추출되지 않아 에러 정의 파일이 search로 발견 불가 → grep 우회 4~5턴. 설정 기본값(포트 등 숫자)·에러 메시지 같은 문자열 리터럴도 인덱스에 없었다.
- **수정**: Rust `enum_variant`를 kind `variant`로 추출하고 owner(소속 enum)를 부여. tantivy `literal` 필드 신설(부스트 1.0 — 심볼 4.0·독스트링 2.0보다 낮게, 256자 캡). 포맷 버전 범프로 재인덱스 강제.
- **검증**: 에러 정의 파일이 search 1위, t5류(기본 포트)가 첫 search 응답에서 발견.

### 1.5 발견 — caller 스캔 이름별 예산 (callers.rs)

- **관찰**: `scan_cap`(500)이 전 스캔 이름 공유의 전역 예산이라, walk 초반의 핫한 이름이 소진하면 뒤쪽 파일의 호출처가 0건 — 실제 호출처 19곳이 있는 `put_tb`가 "no direct caller observed"로 출력됐고 절단 표시도 없었다.

- **검증**: t2(정의+호출처)가 search 1콜에 통째로 실린다.

### 1.6 안정성 — 동시 기동 인덱스 wipe 경쟁 (index.rs)

- **관찰**: 신규 repo에서 서버 2개가 동시 기동하면 둘 다 포맷 sidecar 부재 → wipe+재생성 경로를 타며 **서로의 인덱스 파일을 삭제**했다 (`test_mcp_parallel_requests` 플레이크의 원인).
- **수정**: 빈 인덱스 디렉터리(meta.json 없음)는 wipe 없이 제자리 생성, 파괴적 복구 전 1회 재시도.

### 1.7 무효 회차 v4 — 프롬프트 재구성 사고

v4는 프롬프트 재구성 실수로 도구 유도 문구가 빠져 search가 182콜 중 3콜만 쓰인 오염 회차다(측정 무효). "프롬프트는 tasks JSON에서 jq로 추출, 한 글자도 수정 금지" 규칙(workflow §4-2 실측 함정 표)이 여기서 나왔다.

## 캠페인 2 — ollama + ClickHouse (C/C++): 2-arm 160 에피소드

claude sonnet + codex gpt-5.5의 2 arm × 10과제 × 2rep × 2repo = 80 에피소드/회차. 멀티 모델 오케스트레이션(fable 계획·검증 / sonnet 측정 / opus 수정) 첫 도입.

### 2.1 측정 전 게이트 — 참조 반환 심볼 3,053줄 누락 (parser.rs, 블로커)

- **발견 경위**: 벤치마크 시작 전 fable의 최종 검증이 실저장소 인덱싱 스모크로 적발했다 — opus 적대 리뷰 **2회를 통과한** 코드였다. tree-sitter-cpp의 `reference_declarator`는 `declarator` 필드가 없는(positional child) 노드라 필드 기반 탐색이 항상 None → `int& getref()`, `T& operator=` 류가 통째로 미추출. ClickHouse src 기준 정의 라인 약 3,053개.
- **의미**: 수정 없이 측정했다면 ClickHouse 과제 다수가 "제품 결함인지 검색 한계인지" 분리 불가인 채로 오염됐다 — "측정 전 실저장소 인덱싱 스모크" 게이트가 여기서 굳어졌다. 같은 게이트에서 함수-로컬 vexing-parse 가짜 심볼(전체 심볼의 8.5%)과 in-class private 인라인 정의의 exported 오판도 함께 수정 → 포맷 `v6`.

### 2.2 iter1 관찰 3건 → 수정 → iter2 검증 (77/79 → 80/80)

- **grep 기본 output_mode** — 관찰: grep 호출의 절대다수가 기본값 (files_with_matches)을 그대로 사용해 줄 번호 없는 파일 목록만 받고 search/read 재탐색 루프에 빠졌다(비율 수치는 workflow §9-9 교훈 2). 최악 사례 c7-r2(INFINITE_LOOP 에러 코드 정의 + throw 지점 7곳)의 전사가 이 실패의 해부도다: grep 9회 전부 기본 모드(파일 목록만) → 줄 번호를 못 얻자 search 25회 표류(BM25가 무관한 ASTCreateWasmFunctionQuery.h 류를 상위에) → read의 start_line/end_line이 인식되지 않아 109,553B 전체 파일 읽기를 시도하며 -32602 오류 19회 → 도구 호출 49회(319s) 끝에 러너 컨텍스트 한도("Prompt is too long")로 미완성. 같은 과제 c7-r1은 `output_mode="content"`를 명시한 grep 1회로 모든 줄 번호를 즉시 얻어 2턴에 끝났다 — 같은 도구, 인자 하나 차이. 캠페인 1 교훈("줄 번호가 없으면 이미 본 범위를 다시 읽는다")이 grep에서 재현된 것. 수정: 기본값을 content(+줄 번호)로 전환(grep.rs L132-133). 검증: c7이 컨텍스트 한계 partial에서 두 rep 모두 **3~8턴 correct**(throw 지점 7곳 전부 열거)로, grep 평균 응답 바이트 4,095 → 1,690. 에이전트는 압도적으로 최소 인자로 도구를 부른다 — 기본값이 가장 뜨거운 코드 경로다.
- **read 관용 파라미터** — 관찰: claude 계열이 `path`/`file`/`start_line`/ `end_line`을 관용적으로 보내 52건 하드 에러(codex는 0건 — 스키마 준수 성향이 모델별로 다르다). Claude Code 내장 Read 도구의 파라미터 습관을 그대로 가져오는 것이다. 1차 수정은 별칭 수용(read.rs L39). 검증: iter2에서 에러 52 → 5건. 잔여 원인은 2.3 참조.
- **owner 토큰 인덱싱** — 관찰: `get`처럼 짧고 흔한 이름은 BM25 식별력이 없어 "StorageFactory get" 한정 쿼리로도 미발견 — c8-r1은 search 10회가 전부 정답인 `get` 대신 다른 멤버의 `instance()`나 헤더 선언만 띄웠고, 최종 답변은 헤더 선언만 인용한 partial로 끝났다(19턴, 95,057B). 수정: 멤버 심볼 문서에 owner 이름·분리 토큰을 추가 인덱싱(`v7`). 검증: c8-r1이 **search 2회 즉시 발견**, 6턴 correct(28,722B).

### 2.3 오진과 정정 — read 결함의 진짜 근본 원인

- **iter2가 남긴 잔존 결함 4종**: ① read `query` 별칭 미지원(-32602 5건, 전부 재시도 복구), ② `file` 별칭 + start_line/end_line 동시 사용 시 줄 범위 무시 — 파일 전체 반환으로 102,400B 한도 초과 연쇄(c7-r1에서 4건, StorageDistributed.cpp 109,553B 등), ③ `start`/`end` 별칭 미처리(o5-r1, 196,826B 전체 반환 3회),
  ④ overview의 대형 파일 102KB 한도 초과(o9-r1·o5-r1). 전부 재시도로 복구돼 정답률 영향은 없었지만 턴·바이트를 낭비하는 효율 결함이었다.
- **오진**: sonnet의 iter2 분석은 이들을 "별칭 미지원" 문제로 진단했다 — 별칭 표만 넓히면 끝난다는 처방.
- **정정**: fable의 라이브 바이너리 검증에서 ②③의 진짜 공통 뿌리가 드러났다 — 에이전트가 숫자를 JSON 문자열(`"228"`)로 보내면 `as_u64()`가 **조용히 None**을 반환해 줄 범위가 통째로 탈락, 파일 전체가 렌더링됐다(102KB 한도 오류 연쇄). 오진대로 별칭만 추가했다면 최종 수정 라운드는 아무것도 고치지 못했다.
- **수정**: 공용 `lenient_usize` 코어션 + 별칭 맵으로 일반화. **별칭 두더지잡기가 아니라 코어션 레이어가 올바른 설계** — 이 캠페인의 핵심 도구 설계 교훈이며, 그 완성은 캠페인 5 루프 2의 별칭 키 정규화다(5.3).
- 같은 정정 패스에서 first_answer_turn 하락(0.8→0.4)을 회귀로 해석한 것도 뒤집혔다 — ToolSearch(하니스 메커니즘)를 도구 호출로 센 메트릭 아티팩트였고 보정 시 0.9→0.9 동일. **점수는 전부 정직했고 서사만 틀렸다.**

### 2.4 모델 역할에서 얻은 것

sonnet은 현상 발견(grep 기본값·read 에러)에 정확했지만 코드 수준 root-cause에서 사실 오류를 냈고, opus는 정확한 진단이 주어졌을 때만 정확했으며, 블로커 적발과 오진 정정은 모두 fable 게이트에서 나왔다. 여기서 "물량은 sonnet, 명세된 수정은 opus, 진실 판정은 fable" 배치와 "정정 패스를 분석 직후에 둔다" 규칙이 workflow §3(모델 배치)·§4-7(분석·검증 사다리)로 굳어졌다. 효율 검증 핵심값: claude 중앙값 턴 −53%(9.5→4.5), 응답 바이트 −51~58%(상세는 `benchmark-workflow.md` §9-5).

## 캠페인 5 — 응답 다이어트 (ds-iter2 → ds-iter3): 에이전트는 보상한다

### 5.1 루프 1 (ds-iter2) — P1~P6 렌더 수정

ds-base1 분석에서 도출한 렌더 계층 수정 6건(인덱스 포맷·스캔 예산·랭킹·코어션 무접촉): ① 쿼리 매칭 심볼 앵커 렌더링(Tier-1 통식별자 정확 일치 우선, 비매칭 컨테이너는 2~3줄 요약), ② 동일 이름 caller 블록 dedup("same as above"), ③ 정의 ≥5 이름 caller 생략(`caller_omit_def_threshold`), ④ 꼬리 목록 50→12, ⑤ grep 기본 제외에 `.yarn/`·minified 번들 추가, ⑥ 도구 설명에 재확인 불필요 예시.

적용 전 라이브 A/B(보존한 수정 전 바이너리와 동일 쿼리 비교)에서 결함 2건을 사전 적발·정정했다: (a) 토큰 교집합 매칭 과완화 — `select` 부분 토큰이 `get_select` 등에 풀 스니펫 부여 → Tier-1 통식별자 일치 우선으로 정밀화, (b) caller dedup이 스캔 순서 기준이라 원본 블록 없는 dangling "same as" 발생 → 렌더 시점·렌더 순서 기준 + 실출력 후에만 참조 허용으로 이동.

### 5.2 ds-iter2 검증 — 응답은 줄었는데 claude는 보상했다

- **관찰**: 동일 쿼리 A/B 절감은 실현됐고 codex는 에피소드 총량까지 순개선했는데, **claude 에피소드 총량은 오히려 +16% 역행**했다.
- **도구별 분해가 원인을 드러냈다**: search 합은 거의 그대로인데 read·grep 바이트가 급증 — 앵커링이 한 줄 스텁으로 강등한 비매칭 심볼 맥락을 claude가 통파일 read로 다시 가져왔다(최악 과제의 최대 응답이 search에서 통파일 read로 교체). 도구별 분해·과제별 폭주의 상세 수치는 `benchmark-workflow.md` §9-8.
- **교훈 8**(`benchmark-workflow.md` §9-9의 설계 교훈 7가지에 잇는 8번째): 캠페인 1의 "풍부한 응답이 턴을 늘린다"(1.2)의 역방향 짝 — **빈약해진 응답은 보상 read를 늘린다.** 응답 크기 최적화는 응답 단위가 아니라 에이전트의 보상 행동까지 포함한 에피소드 총량으로 판정해야 한다.

### 5.3 루프 2 (ds-iter3) — 보상의 3개 기전을 흡수

fable의 transcript 재진단이 iter2 회귀를 3개 기전으로 분해했고, 셋을 묶어 수정:

1. **시그니처 축약**(s8형 — 스텁 맥락 손실→search 반복): 강등 심볼을 한 줄 스텁 대신 Tier-2는 시그니처+최대 3줄, 비매칭은 시그니처 1줄 + 생략 마커로 렌더.
2. **read 별칭 키 정규화**(s7형 — 제품 결함): `startLine`(카멜)이 별칭 미등록으로 조용히 무시돼 범위 read가 통파일 렌더로 비대해졌다. 별칭 매칭을 키 정규화(소문자화+`_`/`-` 제거)로 일반화 — 2.3 코어션 교훈의 완성.
3. **파일당 앵커 풀 스니펫 상한**(d9형 — `save`·`send` 같은 흔한 이름 다발 앵커): `search_anchor_snippet_limit`(기본 3) 신설, 초과 앵커는 축약으로 강등.

구현 중 A/B로 추가 적발: 심볼 초과 **폴백 분기**에 앵커 캡·축약이 미적용이라 d9가 개선 0이던 것을 양 분기 공유 렌더 함수로 통일(d9 프로브로 확인).

**검증**: claude 에피소드 총량이 **19.5KB로 회복** — iter2 역행을 넘어 다이어트 전 대비 순절감(read 보상 흡수), codex 절감 유지, 정답률 80/80. 유일한 역방향은 claude first_answer 평균의 상승 드리프트(전건 도달) — 차기 추적 항목. 도구별 복귀·과제별 회복 폭의 상세 수치는 `benchmark-workflow.md` §9-8.

### 5.4 사후 적발 — 도구 설명의 d7 정답 단편 누출 (오염 포렌식)

- **관찰**: 캠페인 종결 후 "패치가 벤치마크 특정적인가" 점검에서, 루프 1의 P6이 넣은 도구 설명 예시가 **d7 rubric 4단계 정답(파일·줄·시그니처)과 verbatim 일치**함을 적발. 출처는 오케스트레이터가 d7 진단 사례를 발주 브리프 예시로 그대로 옮긴 실수 — iter2·iter3 측정 내내 모든 search 도구 설명에 노출됐다.
- **포렌식**: 세 회차의 d7 에피소드 전수에서 정답 줄 번호가 답변 이전에 도구 결과에 먼저 등장 — 맹목 인용(힌트 베끼기) 0건, 전부 자력 발견.
- **정정·재검증**: 예시를 중립 문구로 교체(렌더 바이트 무변화, 전 과제 정답과의 충돌 전수 검사 통과) 후 무힌트 재실행(ds-d7check): 전건 correct, 함정 0 — **정확도 결론은 오염 무관**. 그러나 claude의 무힌트 d7 턴이 힌트 회차보다 길게 복귀 — **d7 턴 개선은 렌더 효과가 아니라 힌트 효과였다.** 보정 결과 iter3 claude의 "턴 소폭 개선" 주장은 철회됐고, 평균 바이트 절감폭은 오히려 확대(−22.5%)됐다. 포렌식 전수 수치·보정 집계는 `benchmark-workflow.md` §9-8.
- **교훈**: **제품이 contestant에게 출하하는 모든 문자열(도구 설명·에러 메시지· 예시)은 측정 표면이다.** tasks JSON의 expected file/line/symbol과의 기계 diff가 워크플로우 불변 조건(`benchmark-workflow.md` §4-1 원칙 6)으로 추가됐다.

## 측정 도구도 결함을 낸다 — 리팩터링 회귀 차분과 다중 세션 규율 (2026-06-14)

구조 리팩터링(god-module 해체·`lang/` 레지스트리 도입, 9개 child)의 **행위 보존**을, contestant 벤치가 아니라 **OLD(`33cc7eb`)/NEW(HEAD) 바이너리의 도구 출력 바이트 차분**이라는 결정적 도구로 검증하려 했다 — 포화된 세트에서 정확도는 변별을 못 하고(§캠페인 3·4), 행위 보존이면 도구 출력이 바이트 동일해야 하므로 차분이 더 직접적·국소적이다.

- **1차 실패 — 약한 측정 장치가 거짓 회귀를 만든다**: 측정 장치(정규화·러너)를 sonnet이 만들자, caller/callee·BM25 랭킹 렌더에 남은 Tantivy/HashMap 순회 비결정성을 정규화가 다 잡지 못해 django·ollama가 OLD≠NEW로 나왔고 "실제 회귀"로 오판될 뻔했다. ClickHouse 담당만 **OLD-vs-OLD 통제**(같은 바이너리를 두 번 fresh 인덱싱한 diff가 OLD-vs-NEW diff와 바이트 동일)로 비결정성임을 입증했다. **교훈①: 측정 장치(apparatus)도 측정 대상이다** — 가장 강한 모델로 짓고(이 회차에서 sonnet→opus로 정정), "같은 바이너리 2회 diff=0"을 참 결정성의 합격선으로 강제하며, OLD-vs-OLD/NEW-vs-NEW 통제로 비결정성과 실제 변화를 분리한다. 보강 증거로 호출자 렌더 문자열이 OLD/HEAD 소스에서 hit 수 완전 동일이라 잠정 판정은 회귀가 아니라 비결정성이며, 결정적 장치로 재실행해 확정한다.
- **교차 모델 적대 리뷰가 측정 타당성 결함을 잡는다**: opus가 장치를 재구축하고 **codex(gpt-5.5, `model_reasoning_effort=xhigh`)**가 장치를 적대 리뷰하자, codex는 다수 블로커 후보를 적발했다 — 경로 sort가 진짜 랭킹 변화를 가리는지, rank-100 검색 cap 포화, `find` 순서 결정성, 기본 cap truncation 통지 등(codex의 주장이며 일부는 오탐일 수 있어 opus가 건별 검증 후 반영한다). 측정 장치는 단일 모델 자기검증이 아니라 다른 CLI·모델의 red-team을 거쳐야 한다(효과가 입증되면 contestant 캠페인의 데이터셋·하니스 구축에도 적용 검토).
- **교훈②(핵심) — 다중 repo·다중 검사는 다중 세션으로**: 그 개선 단계를 **단일 세션이 codex 블로커 6건 수정과 ClickHouse+ollama 다중 repo 재검증을 겹쳐 처리**하게 짜자 ~1h로 늘어졌다(transcript ~563KB; 구조 페널티의 정확 분해는 미측정). 공유 러너 편집만 단일 세션이면 되고(단일 writer + 부트스트랩 repo 1개로 결정성 확인), repo별 결정성 검증·diff는 **단위마다 별도 세션으로 fan-out**해야 한다 — 실제로 그렇게 짠 diff 단계는 깨끗했다. 이는 `benchmark-workflow.md` §4-1 #7 세션 입도 규칙으로 규범화됐고, "에피소드별 에이전트 스폰 금지(실행은 스크립트로)"(#3)와 짝을 이룬다: #3이 '너무 잘게 스폰 말라'면, #7은 '한 세션에 너무 많이 몰지 말라'다.
- **교훈③(후속) — 수렴 루프는 유계로, 횟수는 스크립트가 집행, 안 되면 합동 에스컬레이션**: fan-out으로 교정한 뒤에도 "통과할 때까지"를 맡은 단일 improve 세션이 ClickHouse caller cap의 *환원 불가* 비결정성을 사후 정렬로 잡으려다 **정지 시점까지 battery ≥86회·~37분** 그라인드했다 — per-name cap에 걸린 호출자 목록은 매 run마다 cap을 통과해 **표시되는 항목 집합**이 HashMap 순서로 달라지는 것으로 관찰됐고(순서가 아니라 나타나는 항목이 다름), 그래서 정렬로는 0을 만들 수 없다. 옳은 처방은 **잘린 caller/callee 본문을 diff에서 제외**하되 개수·cap 메시지·비잘림 섹션·추출 내용은 유지·비교하는 것(안정 영역과 개수에서 실제 변화는 여전히 포착 — 마스킹 방지). 핵심은 **횟수를 에이전트 프롬프트가 아니라 워크플로 스크립트가 집행**해야 한다는 것이다 — "3회까지" 문장만으로는 그라인드하던 그 세션이 멈출 판단을 못 한다. attempt마다 별도 에이전트 1회(편집+검증)로 쪼개 스크립트가 ≤3 집행, 미수렴 시 opus+codex(gpt-5.5 xhigh) 합동 재검토 1회 후 그래도 안 되면 정지 — `benchmark-workflow.md` §4-1 #9로 규범화. caller cap 출력 불안정 자체는 제품 측 항목으로 백로그 10번에 별도 기록했다.

## 캠페인 6 — deno (재구조화 제품 기준선): 포화는 유지되고, 변별은 d10 한 곳 (2026-06-14)

직전 절의 구조 리팩터링이 **행위 보존**임을 결정적 차분으로 확인한 게 (1)이었다면, 같은 `refactor/cms-restructure` HEAD를 캠페인 1~5에 없던 **신선한 코퍼스 deno**(Rust/TS/JS)로 측정해 재구조화 제품의 현행 효율 기준선을 뜬 게 (2)다. 회귀 검증이 아니라 제품 개선용 기준선이라 개선-rerun 없이 단일 패스 1회로 종결했다. 수치 정본은 `benchmark-workflow.md` §9-14; 이 절은 인과만 남긴다.

- **포화가 또 유지됐다.** 39/40 correct + partial 1, wrong 0. 한계 5(포화된 등급에서 정답률은 변별하지 못한다)가 신선한 Rust/TS에서도 성립했다. 유일한 비-correct는 d10(모호한 자연어 개념 — "CLI 모듈 로더의 `cjs_tracker` 타입")의 claude rep2: search 2회·read 0회로 원본을 열지 않고 절단된 답을 내 정답 타입을 줄 번호로 인용하지 못했다. **§9-13이 예고한 "모호한 자연어 개념 쿼리"가 실제로 변별한 첫 사례지만 폭은 1/40**이다 — 변별 가능한 과제 세트 백로그는 여전히 열려 있다.
- **게이트가 두 번 일했다 — 데이터셋 결함을 작성자(opus)가 아니라 교차모델·contestant가 적발.** (a) **codex gpt-5.5 xhigh 적대 리뷰**가 매트릭스 전에 d6(`CheckRunPermissionError` 혼동)·d9(누락 사용처 `cli/lib.rs:972`)·d10(`CjsTracker` 형제 타입 모호)을 잡아 좁혔고, (b) **warmup**이 d1의 `DEFAULT_PORT` 2중 정의(`cli/args/flags.rs:5915` CLI 파서 vs `ext/node/ops/inspector.rs:64` node op — 작성 시 `head` 절단으로 놓침)를 contestant가 두 곳 인용하면서 드러냈다. 직전 절 교훈①("측정 장치도 측정 표면")이 **데이터셋에도** 적용됨을 재확인한다: ground truth는 opus가 rg로 짓되 교차모델 red-team과 warmup 게이트를 통과해야 한다.
- **효율 프로파일이 미접촉 코퍼스에 전이.** claude는 search 주도·중앙값 4턴·first-answer≈1.2의 한 방 검색 구조, codex는 read 중심(read 115)·중앙값 7턴의 grep+read 검증 구조 — 캠페인 3·4의 CLI별 스타일 분리가 신선한 Rust 코드에서 그대로 재현됐다.
- **contestant 환경의 도구 카탈로그 확장 — 오염 가설을 기각하고 purity 정의를 재확인.** 설치된 claude CLI가 내비게이션과 무관한 확장 도구(scheduling/task/worktree 계열)를 기본 노출해 부모 오케스트레이션 세션의 env 누출을 의심했으나, **클린-env 프로브**(알려진 CLAUDE_CODE_*·CLAUDECODE 스크럽)에서 도구 목록이 불변이어서 부모 세션 누출 가능성을 낮췄다(best-effort 확인이며 완전 증명은 아니다 — 6개 변수만 스크럽). 결과 타당성에는 무영향이다: 파일/셸 도구는 전부 차단되고 `denied_builtin_attempts`가 전건 0이라 navigation 정답에 활용 불가. **교훈: pure 격리의 보증은 "도구 목록이 비었다"가 아니라 "비-MCP 도구 사용 0"이다** — 카탈로그가 비지 않아도 사용 0이면 pure이고, 그 사용량을 에피소드별로 기계 검증(`verify-metrics.sh`)한다.

## 채점 신뢰성 — 재채점 게이트 4회가 말해주는 것

매 회차(ds-iter1·ds-base1·ds-iter2·ds-iter3) 동일 구성의 표본 40건 — 어려운 과제 8종(d6~d9, s7~s10) + 중간 2종(d10, s5) × 4에피소드 — 을 "sonnet ×10이 인용 줄 전수 원본 대조 + rubric 체크리스트를 기계 수집(`evidence.json`) → fable이 플래그 판정"하는 분업으로 재채점했고, 쟁점 에피소드는 fable이 sed 원본 대조로 별도 독립 검증했다. 수집과 판정을 분리하면 물량을 키우면서도 게이트 품질이 유지된다 (workflow §4-7). 이 절이 게이트 4회의 판정 기록을 통합한 단일 출처다.

- **ds-iter1 / ds-base1 / ds-iter2: overturn 0.** 인용 대조 ~300건 수준에서 나온 불일치 플래그는 전부 rubric 비요소로 채점 비영향 판정. 플래그의 결 자체가 방법론의 신뢰 근거다:
  - **의역·표기 축약**: s5-r1의 cron.ts:30-32 키 이름 의역(답안 `taskExpression` vs 원본 `key` — 보조 인용, add 본문 인용은 정확), d10의 `class Q:` 축약 표기(원본은 베이스 클래스를 포함한 전체 시그니처 — 위치 주장은 정확). 전부 rubric 요소 충족.
  - **줄 라벨 ±1 전사 드리프트**: s8의 마크다운 스니펫 줄 라벨 ±1~3(빈 줄 생략로 드리프트) — 도구가 반환한 줄 번호는 전 건 정확했고, ±1은 모델의 전사 습관이다(줄 라벨 ±0 rubric을 만들면 변별 요소가 될 수 있음). s7의 한 호출 지점 줄 라벨 ±1(호출 표현식 줄과 그 인자 리터럴 줄이 1 차이)은 3회차 연속 같은 패턴으로 플래그되고 같은 사유로 기각됐다 — 수집기의 과민함이 회차를 가로질러 일관적이라는 것 자체가 수집이 기계적이라는 증거다.
  - **형식 문제**: codex-base arm의 절대 경로 마크다운 링크 표기(상대 경로 정보 병기, 위치는 정확) — 채점 비영향.
  - **수집기의 요건 과잉 해석은 판정에서 기각**: ds-iter2 s9-r2에서 수집기가 unsatisfied로 표시한 "재귀/외부 호출 구분 서술"은 rubric의 함정 해설이지 채점 요건이 아니다 — fable이 rubric 원문 대조로 기각(ds-base1 동형 사례와 일관).
- **ds-iter3: 시리즈 최초 overturn 2건 (partial→correct)** — 둘 다 모델 답안 회귀가 아니라 **1차 채점자 오류**였다. s9-r2는 답안에 명백히 존재하는 섹션(해당 줄 인용 + 스니펫)을 누락으로 판정한 사실 오인, s10-r1은 rubric의 "추가 언급 감점 없음" 조항 미적용. 두 건 모두 "채점자가 더 엄격" 방향이고, 같은 답안 스타일이 이전 회차들에서는 correct였다 — 즉 채점 변동성이다.
- **deno-b1 (캠페인 6): overturn 0.** 별도 캠페인의 5번째 게이트. 표본 구성은 hard 과제(d8·d9·d10) 전 에피소드 12건(비-correct 전건 포함) — sonnet 3명이 인용 file:line을 deno 원본과 전수 대조(rg/Read 71회, 측정 도구 미사용) → opus 판정. 12/12에서 채점자 판정이 원본과 일치, partial 1건(d10 claude rep2, 절단된 답이 정답 타입을 줄 번호로 인용 못 함)도 확정. 부수 관찰: contestant들이 expected에 없던 실제 정의(d9 `cli/snapshot/lib.rs:4`의 `pub static CLI_SNAPSHOT`)까지 찾아냈으나 rubric은 사용처 ≥1곳을 요구해 점수 무영향 — d9 expected가 정의를 빠뜨린 보강 여지로 기록.
- **교훈**: 기계 판정 가능한 rubric + 수집·판정 분리는 세 캠페인 연속(이제 deno-b1 포함 네 게이트) 불일치 0을 만들 만큼 견고하지만, 채점자 엄격성 편차까지 막지는 못한다(ds-iter3의 2 overturn). → **1차 채점 rationale에 rubric 조항 인용을 의무화**(차기 채점 프롬프트 보강 항목).

## 백로그 — 캠페인이 남긴 제품 개선 항목

연원부터 정리한다. 캠페인 1이 남긴 1차 백로그 5항목 중 **리터럴 줄 번호, Java/Kotlin enum variant 추출, find 글롭 설명 보강은 HEAD에 구현 완료**가 코드로 확인됐고(`ExtractedLiteral`+`[L{n}]` 렌더, `enum_constant`/`enum_entry` 캡처, basename 캐비엇), **타 언어 저장소 검증은 캠페인 3 그 자체**였으며, 다단계 흐름 추적은 아래 9번으로 재평가됐다. 아래 목록은 2026-06-12 전수 분석 — 캠페인 2 전사 160 에피소드 정밀 판독(search 281건 분류) + HEAD 라이브 재현(django/strapi) + dogfooding 인스턴스 직접 호출 — 이 식별한 후보들로, 이 중 동일 이름 caller 중복 압축(반복 50건, annotation이 search 바이트의 ~17%)은 캠페인 5 루프 1의 P2·P3로 이미 반영됐다. 잔여 항목을 우선순위순으로 정리한다.

1. **쿼리 토크나이즈 정합화** — `StorageFactory::get`·`QuerySet.__iter__`처럼 구두점이 붙은 쿼리어를 비식별자 문자(`::`, `.`, `/`)로 추가 분해해 BM25 토크나이저·매치 선택 로직과 정합시킨다. 캠페인 2 search 281건 중 20.6%가 구두점 단어였고(c8-r1 fallback 추락의 한 축), HEAD 라이브 재현에서 django(`QuerySet.__iter__ definition` → 정답 6위 밖, 공백 분리형 → 1위)· strapi(`strapi.documents`) 모두 글루드/분리형 랭킹 역전이 확인됐다.
2. **EXACT_NAME 부스트의 흔한 이름 게이트** — `constructor`(11자)·`Definition` 같은 글루 단어가 "8자 이상 = 식별력" 규칙을 통과해 ×3.0 부스트로 무관 파일을 1위로 끌어올린다. 캠페인 2 o6(같은 이름 정의 28개의 `constructor` 스니펫만 17,276B, 정답은 응답에 부재)·c4-r2·o9-r1이 실측 사례 — 저장소 내 정의 빈도(doc_freq) 게이트로 부스트·승격을 차단한다(매치 자체는 유지).
3. **fallback 심볼 목록의 쿼리어 겹침순 정렬** — fallback이 줄 순서 상위 20개에서 절단돼 정답 심볼이 숨는다. 캠페인 2 o5-r1에서 정답 3심볼이 정확히 "9 more symbols not shown" 안에 있어 overview+grep+read 6턴·73,724B를 추가 소모했고, 캠페인 1의 "정답 심볼 `put_tb`가 목록에 안 보임"(1.1)과 동일 기전이다. 캠페인 5의 앵커 렌더링·폴백 분기 통일이 같은 증상의 상당분을 렌더 측면에서 흡수했을 수 있으므로 재측정 후 잔여분만 적용.
4. **긴 자연어 쿼리의 부분 일치 임계 캡** — 부분 승격 임계 `div_ceil(n/2)`가 쿼리 길이에 비례해 올라 6단어+ 자연어 쿼리에서 사실상 승격 불가. 캠페인 2 iter2 무스니펫 search 14건 중 9건이 5단어+ 쿼리였고, o3-r2는 정답 심볼이 3개 단어를 맞추고도 탈락해 search 3연속 실패 후 grep 4회로 우회했다. `min(⌈n/2⌉, 3)` 캡 — 단 이름 증거 게이트(1.2)는 유지해 32KB 비대 회귀를 막는다.
5. **grep 코어션 레이어** — 캠페인 2의 "코어션 레이어가 올바른 설계" 교훈(2.3)이 read에만 적용된 잔여분. 양 캠페인 grep 276건 실측에서 `file_path`/`file`/ `query` 등 path 별칭류 22건이 조용히 무시돼 의도치 않은 저장소 전역 검색이 됐고, 컨텍스트 별칭(`context_lines`/`C`/`n`) 31건도 무시됐다. claude arm 전용 효과(codex는 스키마 준수 100%).
6. **리터럴 exact-value 승격의 글루 단어 게이트** — 단일 쿼리어가 리터럴 전체와 일치하면 무조건 표시하는 규칙 때문에 캠페인 2 iter2 응답의 리터럴 줄 275건 중 78%가 8자 이하 단일 영단어 노이즈(`"route"`, `"port"`)였다. 식별력 조건(숫자· 기호 포함 또는 다단어)을 달면 본래 표적(포트 등 숫자 리터럴, 에러 메시지)은 통과한다 — 리터럴 줄 번호 기능(캠페인 1 백로그)이 구현되며 생긴 부작용의 후속.
7. **식별력 있는 숫자 리터럴 선별 인덱싱** — 따옴표가 있는 숫자 문자열 리터럴은 search 1위로 잡히는데 따옴표 없는 동일 숫자는 미인덱스 — 차이는 따옴표 유무뿐이다. HEAD 라이브 검증에서 따옴표 없는 포트 숫자 검색은 상위가 전부 rubric이 함정으로 지정한 테스트/URL 파일이었다. 3자리+ 숫자를 기존 `literal` 필드에 최저 부스트로 추가(재인덱스 1회).
8. **인덱스 비호환에 대한 런타임 자기복구** — 상주 구버전 서버 아래에서 신버전 바이너리가 인덱스를 재생성하면 search가 `IncompatibleIndex`로 영구 다운된다 — dogfooding 인스턴스 **5/5에서 실측**. 복구 경로는 기동 시에만 존재하고 런타임 search 오류는 -32603 Rust 에러 원문 그대로라 에이전트가 다음 행동을 정할 수 없다. 벤치 지표 효과는 없지만(에피소드는 매번 새 프로세스) 실사용 신뢰성 결함이자 향후 dogfooding 측정의 전제 조건이다.
9. **다단계 흐름 추적 — depth-2 callee (조건부 후순위)** — t4류 "RPC 분기 → 실행" 4단계 체인이 캠페인 1 전 회차에서 25~33턴(타 과제 중앙값 7~9턴의 3~4배): depth-1 callee 주석으로는 체인이 접히지 않아 에이전트가 단계마다 search/read로 전개한다. 설계 검토의 결론 — (a) **디스패치형 함수(depth-1 callee ≥3 또는 match/switch 다분기) 한정 depth-2 확장**: 추가 워크스페이스 스캔 없이 disk read N회(≤`callee_list_cap`)로 비용은 최소지만, 이름 매치 오차가 depth마다 누적되므로 `(approximate, via {callee})` 레이블이 필수고 그 레이블·바이트 비용이 핵심 트레이드오프. (b) 시작→끝 **호출 체인 요약 응답**: 에이전트 경험 개선은 최대지만 호출 그래프 온-디맨드 구성의 첫 호출 지연(파일 수 비례 disk read)과 구현 범위가 가장 크다. (c) 현행 유지. 권장은 (a)의 조건부 파일럿 — 단 2026-06-12 재평가는 flow 과제 잔여 비용의 지배 요인이 callee 깊이가 아니라 "진입점 심볼이 detail로 노출되지 않는 것"(위 1~4번의 표적)임을 전사에서 확인했다. 1~4 적용 후에도 flow 과제 턴이 중앙값 2배를 넘으면 착수, 아니면 종결한다.
10. **caller/callee per-name cap 출력의 결정성** — per-name hit cap에 걸린 호출자 목록은 같은 입력·같은 바이너리에서도 인덱스 빌드마다 cap을 통과해 *표시되는 항목 집합*이 HashMap 순회 순서로 달라진다(2026-06-14 회귀 차분에서 측정 장치가 same-binary diff=0을 못 만든 원인 — `측정 도구도 결함을 낸다` 교훈③). 출력이 `list may be incomplete`로 불완전을 고지하므로 정답률 결함은 아니나, 같은 질의에 caller 목록이 run마다 바뀌는 것은 재현성·신뢰성 저하다 — cap 선별을 결정적 기준(파일 경로·줄 번호 정렬 후 상위 N)으로 바꾸면 해소된다. 측정 장치 회귀 차분의 부산물로 발견.

## 맺음말 — 남은 한계

- **과제 세트 포화 3연속**: C/C++ 셋, 일부러 어렵게 만든 django+strapi 셋, baseline arm까지 전부 100% — baseline이 100%인 세트에서는 "빌트인이 실패하는 곳에서 codemap이 돕는가"를 보일 수 없다. 정확도 축의 변별력 확보(depth-2 callee, 모호한 자연어 개념 검색, 정답 분산 과제, 줄 라벨 ±0 rubric, 5,000줄+ 파일)가 측정 백로그 1순위다. 프롤로그의 미검증 변수 — 대형 코퍼스 교차점, 인터랙티브 main-context — 도 여전히 열린 질문으로 남아 있다.
- **t4류 다단계 흐름 추적**: depth-1 caller/callee로 hard 6~9턴까지는 내려왔지만 체인 자체는 접히지 않는다 — 설계 분석과 착수 조건은 백로그 절 9번.
- **수용된 파서 한계(캠페인 2 종결 시 결정)**: C++ 장꼬리 graceful skip(변환 연산자, in-class `operator new`, 포인터 반환 fn-ptr typedef 일부)과 단일 한정 `ns::func`의 owner 오귀속(빈도 낮음)은 수정하지 않기로 수용했다 — 재발견 시 결함이 아니라 수용된 한계로 취급할 것.
