# [feat] codemap-search line ranges + agent routing

## Work Type
feat

## Current State (As-Is)
- `search` (`apps/codemap-search/src/mcp.rs`) renders matched files and symbols with no line information: the overview branch prints `### {file_path}` headers and `- {name} ({kind})` bullets; the detail branch prints `### File: {file_path}` headers and `- Symbol: {name} ({kind})` bullets plus a source snippet. There is no file total-line count and no per-symbol line span anywhere in the output.
- `get_codemap` delegates rendering to `apps/codemap-search/src/codemap.rs`. The root view (`RootCodemap`) and folder view (`FolderCodemap`) print `- File: {path}` and `  - {name} ({kind})` with no line data; only the detail view (`DetailsCodemap`) already prints a per-symbol `Range: {start_line}:{start_col} - {end_line}:{end_col}`. No view prints a file total-line count.
- `ExtractedFile` (`apps/codemap-search/src/parser.rs:37`) has fields `file_path`, `symbols`, `literals`, `docstrings` — there is no total-line-count field. Per-symbol line spans already exist on every `ExtractedSymbol` as `range: CodeRange { start_line, start_col, end_line, end_col }` (`parser.rs:8-13`, `:27-33`).
- `SearchResult` (`apps/codemap-search/src/index.rs:10-15`) carries `file_path`, `score`, `matched_symbols`, `matched_literals` — no file-level line count to render in `search` output.
- The MCP `initialize` response (`mcp.rs:253-272`) returns `protocolVersion` / `capabilities` / `serverInfo` only — there is no `instructions` field, so the server has no agent-facing guidance channel.
- All five tool descriptions in `tools/list` (`mcp.rs:275-350`) use ordering-framed prose: `get_codemap` = "STEP 1 — orient", `search` = "STEP 2 — locate by keyword", `read`/`find`/`grep` = "STEP 3". This step-order framing pushes agents toward a fixed sequence rather than routing by intent, and coding agents default to `search` over `get_codemap` even when they already hold a path.
- The hierarchical-codemap tool is registered under the name `get_codemap` (`mcp.rs:278` tool-list name, `:453` dispatch arm) and referenced by that exact name in the `search` description (`:290`), the overview footer (`:419`), ~12 e2e test sites (`tests/e2e/tools.rs:53`, `mcp.rs:44`/`:47`/`:174`/`:339`/`:412`, `cross_feature.rs:139`/`:157`/`:211`/`:234`, `scenarios.rs:167`/`:199`), and `README.md` (lines 10, 76, 155). The `get_` prefix and the term “codemap” are inconsistent with the bare sibling tools and unfamiliar to coding agents.

## Desired Outcome (To-Be)
- Every file header in both `search` (overview + detail) and `get_codemap` (root + folder + detail) shows the file's total line count, rendered as `(N lines)` appended to the existing header.
- Every symbol/item bullet under a file shows its line range. New render sites (`search` overview, `search` detail, `get_codemap` root, `get_codemap` folder) use the compact form `[L{start_line}-{end_line}]`. The `get_codemap` detail view keeps its existing richer `Range: {start_line}:{start_col} - {end_line}:{end_col}` form unchanged.
- `ExtractedFile` carries a `total_lines` value populated once at extraction time, so the count flows uniformly to `search` (via the stored `extracted_json`), `get_codemap` (via live re-parse), and the CLI `codemap` command (shared renderer) without per-call recomputation.
- The MCP `initialize` response includes a populated `instructions` field that documents all five tools (`overview`, `search`, `read`, `find`, `grep`) and tells the agent how to route between them (path/structure in hand → `overview`; only a keyword and unknown location → `search`; confirm exact content/lines → `read`/`find`/`grep`).
- The hierarchical-codemap tool is renamed `get_codemap` → `overview` (a bare, intent-clear name consistent with its sibling tools), and the new name appears everywhere it is referenced: dispatch arm, tool list, in-description cross-references, the `instructions` field, the e2e tests, and `README.md`.
- The `overview` and `search` descriptions are rewritten from step-order framing to intent-routing; `read`/`find`/`grep` descriptions keep their "confirm" role but drop the rigid "STEP 3" ordering language.

## Constraints
- `#[serde(default)]` on `total_lines` is mandatory: it lets a pre-existing index (whose stored `extracted_json` lacks the field) deserialize as `total_lines = 0` instead of failing the `serde_json::from_str` at `index.rs:452` and falling back to an empty `ExtractedFile` (which would drop all symbols for that file). `#[serde(default)]` affects deserialization only — every Rust `ExtractedFile { … }` literal must still set the field explicitly or the crate will not compile.
- New render sites use the compact `[L{start_line}-{end_line}]` form; the `get_codemap` detail view's existing `Range: {start_line}:{start_col} - {end_line}:{end_col}` form is intentionally left unchanged (per-view information-density difference is deliberate).
- Line counting uses `file_content.lines().count()`; confirm the value matches what the `read` tool reports for the same file (a file ending in a single trailing `\n` must not be counted as having an extra empty final line) so codemap and `read` agree.
- The `search` output heading `## Codemap overview — {N} matches` (`mcp.rs:399`) is unrelated to the new `overview` tool; leave it as-is unless the implementer finds the wording confusing, in which case a minimal reword (e.g. `## Search overview`) is acceptable but optional.
