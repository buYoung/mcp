## Constraints
- The grammar crate is `tree_sitter_kotlin_ng::LANGUAGE` ŌĆö not `tree_sitter_kotlin`. This name must be used in every `Query::new(...)` call and in code comments. Do not substitute the standard crate name.
- All node names marked `(fixture-confirm)` in the design doc ┬¦8 Kotlin section must be verified against the actual grammar's `node-types.json` (or by running the query against the fixture) before committing navigation.scm. Inspect the `tree-sitter-kotlin-ng` crate's grammar: the `node-types.json` ships with the `tree_sitter_kotlin_ng` crate source (locate it via `cargo metadata`/the vendored crate path under the Cargo registry, e.g. the crate's `src/node-types.json`, or print node kinds by parsing `basic.kt` with the grammar and walking the tree). Confirm the four fixture-confirm items ŌĆö `import_header` path node, `calleeExpression` field name on `call_expression` (the grammar may expose `callee` or another field name), `navigation_expression`/`navigation_suffix` node shape, and `property_declaration` scope boundary ŌĆö against that source. Unverified node names that fail to compile will disable navigation for all Kotlin files at runtime. The confirmed or corrected names must match the committed query file exactly and be documented in a code comment in `navigation.scm`.
- `class_declaration`/interface disambiguation is a required `tags.scm` fixture case: `tree_sitter_kotlin_ng` shares `class_declaration` between `class` and `interface` (distinguished by an `interface` keyword child, as `KotlinSpec::refine_kind` already handles at the symbol layer). The Kotlin `tags.scm` must capture `interface Foo {}` distinctly from `class Foo {}`, and the fixture must assert the two are captured with different `@definition.*` results (e.g. interface as `@definition.interface`/`@definition.type` vs class as `@definition.class`). Do not let both collapse to the same capture.
- Import alias node compile-fail fallback: the aliased `import_header` form (`import com.example.Foo as Bar`) uses an `import_alias`/`type_alias` node shape marked `(fixture-confirm)`. If that alias-node pattern fails to compile against `tree_sitter_kotlin_ng` or does not match the fixture, fall back to capturing the non-aliased `import_header` (plain `@nav.import` + `@nav.import.name`) without the alias capture, document the omission in a `navigation.scm` comment, and keep navigation activated. Alias resolution for Kotlin then degrades to approximate for that import form rather than disabling Kotlin navigation entirely.
- The `tags.scm` file must NOT be included in the runtime concat pass. It is a compile and fixture gate only. The runtime extraction uses `symbols.scm` + `navigation.scm` concat, one `Query::new(...)` + one `QueryCursor` pass per file.
- `navigation: None` vs `Some(NavigationFile { calls: vec![], ... })` semantics introduced by child 02 must be preserved: a Kotlin file with failed navigation extraction produces `None`; a file with successful extraction and zero observed calls produces `Some(empty)`.
- When the snapshot is potentially stale (disk mtime newer than indexed snapshot) or the engine is in warming/error state, `callees.rs` must fall back to the live-disk text scan, not use `navigation.calls` from the stale index.
- `src/lang/mod.rs` is read-only for this child. Per design doc ┬¦8 Stage 8, Wave-6 children do NOT structurally edit `src/lang/mod.rs`; each language activates via a `navigation_query()` override in its own `src/lang/<lang>.rs`. There is therefore no `src/lang/mod.rs` merge hotspot among Wave-6 children ŌĆö child 11 only edits `src/lang/kotlin.rs`.
- child 11 depends on child 06 (`lexical-scope`) having completed. The `LanguageSpec::navigation_query` hook interface must be stable (established by child 02 and confirmed by child 07 pilot) before child 11 begins.
